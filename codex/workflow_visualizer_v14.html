Документация Workflow JSON Visualizer

Назначение Приложение визуализирует JSON workflow engine как rule-driven workflow template. Это не BPMN-редактор и не BPMN-совместимая схема: граф показывает реальные сущности JSON, правила маршрутизации, функции, события и привязки data transform. Цель viewer — помочь быстро понять, как процесс реально будет идти по шаблону, какие условия участвуют в выборе маршрута и где есть внешние хуки или преобразования данных.

Поддерживаемый входной JSON

  • Основная структура берется из processTemplate. Если такого поля нет, приложение пробует использовать template, process или корневой объект.
  • processTemplate.steps — основной список исполняемых шагов. Шаги сортируются по order, а метка берется из marker, name, code или индекса.
  • conditions — библиотека правил. Условия можно найти по id, code, name, title, label или индексу массива.
  • functionMapping — справочник исполняемых функций, из которого берется человекочитаемый код/название функции шага.
  • dataTransforms — справочник трансформаций. Для TransformBinding приложение показывает имя, описание и expression по id.
  • pre/post events и pre/post data transforms читаются непосредственно из каждого шага.

Нормализация JSON

Перед отрисовкой JSON приводится к внутренней модели. Это нужно, чтобы граф не зависел от конкретной формы справочников и мог одинаково работать с массивами, объектами-словарями и неполными ссылками.

TemplateКорневая сущность процесса: имя, версия, тип, processTemplate, справочники условий, функций и трансформаций.
StepNodeИсполняемый шаг из processTemplate.steps. Внутри показываются метка, функция, роль, порядок и служебные состояния вроде WAITING или CONDITIONS_CHECK.
RouteGatewayNodeDecision/gateway из preConditions/postConditions. Название берется из conditions по conditionId; если имени нет, используется functionStatusCode.
RouteEdgeОсновной маршрут процесса между StepNode и RouteGatewayNode. Используется в симуляции токенов.
ExternalEventNodeВнешний preEvent-hook. Визуализируется около шага как аннотация, а не как часть основного исполняемого маршрута.
OutboundEventNodeИсходящее postEvent-событие. Оно отображается для понимания интеграций, но не меняет основной путь процесса.
EventHookEdgeСвязь event-hook с шагом. Рисуется пунктиром и не участвует в token simulation.
EventEmissionEdgeСвязь шага с исходящим событием. Также считается аннотацией.
TransformBindingpreDataTransform/postDataTransform, приклеенный к шагу сверху или снизу. Названия берутся из dataTransforms по id.
WorkflowWarningПредупреждения о пустых conditionIds, неизвестных ссылках на шаги, функции, условия и трансформации.

Правила маршрутизации

  • preConditions и postConditions отображаются как route gateway, потому что это правила выбора маршрута.
  • Condition с id 8203b7df-c2b1-4efd-aec2-195495eccb63 считается Always. Если gateway содержит только Always, он схлопывается в прямую стрелку.
  • postConditions проверяются последовательно: CHECK_NEXT_CONDITION ведет к следующему условию, GO_TO_STEP переводит процесс в целевой шаг.
  • Если postConditions пустой, шаг по умолчанию переходит в следующий шаг по порядку.
  • Альтернативные шаги, события и data transforms визуализируются отдельно от основного маршрута.
  • Подписи стрелок показывают ветку и действие: например true: GO_TO_STEP "160". Для длинных подписей текст переносится на несколько строк.
  • RouteGatewayNode с functionStatusCode без conditionIds не считается ошибкой, потому что сам статус может быть условием маршрутизации.

Визуальные соглашения

  • Обычные шаги с roleId окрашены зеленым, автоматические шаги без roleId — голубым.
  • WAITING и CONDITIONS_CHECK отображаются серым; WAITING показывает SLA в формате WAITING N минут, если значение есть в slaSetting.businessSlaMinutes.
  • RouteGatewayNode рисуется ромбом, потому что это decision point.
  • Data transform приклеивается к шагу: PRE сверху, POST снизу. Tooltip на ? показывает description из dataTransforms.
  • Альтернативные шаги выделяются пунктирной рамкой и выносятся отдельно от основного маршрута.
  • На дальнем масштабе включается overview-mode: мелкие подписи скрываются, чтобы граф оставался читаемой картой.

Правая панель

  • При выборе узла показываются normalized metadata, warnings и Raw JSON в UI-структуре по атрибутам.
  • Для RouteGatewayNode с найденным condition name выводится logic = PLUG AND PRODUCT и conditionBlockList.
  • COMPARISON выводится как label и сравнение left operation right.
  • EXPRESSION и dataTransform.expression показываются как JS-like code с подсветкой синтаксиса.
  • Raw JSON разбивается по ключам и индексам массивов. Простые массивы остаются компактными, чтобы не занимать всю панель без причины.
  • Если у выбранной сущности есть warnings, они выводятся рядом с деталями, а также попадают в нижнюю панель предупреждений.

Симуляция токенов

  • Токены появляются только в режиме симуляции: Play, Step или добавление нового токена.
  • Кнопка ●+ добавляет новый токен поверх уже запущенной симуляции.
  • На обычных шагах токен идет по приоритетному маршруту, а на RouteGatewayNode выбирает исходящую ветку случайно.
  • Симуляция не исполняет JS conditions и не является workflow runtime. Это визуальный слой для исследования маршрутов.
  • В симуляции участвуют только RouteEdge основного процесса. Events, transforms и alternative edges исключены, чтобы токен не уходил в аннотации.
  • Несколько токенов могут идти одновременно. Если они находятся на одном узле, они визуально разнесены вокруг центра.
  • Скорость переключается между 1x, 2x и 4x. Reset очищает токены, а Play создает стартовый токен при необходимости.

Управление

  • Граф можно двигать и масштабировать мышью или кнопками +/−.
  • Поиск по метке срабатывает по Enter и выделяет шаг при полном совпадении без учета регистра.
  • Левая, правая и нижняя панели сворачиваются, а правая панель автоматически раскрывается при выборе элемента.
  • На сильном отдалении подписи скрываются, чтобы граф оставался читаемой картой.

Диагностика и ограничения

  • Viewer показывает структуру workflow template, но не проверяет бизнес-корректность JS expression.
  • Если ссылка на шаг, функцию, условие или data transform не найдена, создается WorkflowWarning.
  • Симуляция не вычисляет conditionBlockList и не вызывает functionMapping. Случайный выбор gateway нужен для визуального исследования возможных веток.
  • Если модель очень большая, часть подписей может скрываться на дальнем масштабе. При увеличении текст возвращается.
  • Приложение полностью локальное: нет backend, сборки, npm, TypeScript или сохранения данных вне текущего HTML.
