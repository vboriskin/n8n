Нужно разработать CLI-приложение на Python для анализа качества управления бэклогом в Jira Scrum-проектах.

Контекст:
- Jira локальная (self-hosted), версия около 8.20.x
- Аутентификация в API будет передана снаружи, через переменные окружения или аргументы CLI
- На вход приложение получает список Jira-проектов (project keys)
- Дефолтный список проектов: SMECM, SMEAM, SMECIS, SMEND
- Использование: аудит качества ведения backlog и планирования у Product Owner / владельцев продукта
- Цель: не оценка людей напрямую, а выявление паттернов проблем в управлении backlog

Нужно реализовать именно CLI-инструмент, а не веб-приложение.

====================================
ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
====================================

Сделай приложение, которое:

1. Принимает параметры запуска:
   --base-url              базовый URL Jira, например https://jira.company.local
   --token                 API token или personal access token
   --projects              список project key через запятую, например SMECM,SMEAM,SMECIS,SMEND
   --since                 дата начала анализа в формате YYYY-MM-DD
   --until                 дата окончания анализа в формате YYYY-MM-DD
   --output-dir            папка для отчетов
   --format                формат результатов: json, csv, html, all
   --verbose               подробный лог
   --config                путь до yaml/json конфигурации
   --max-issues            ограничение на количество задач на проект для отладки

2. Если список проектов не передан, использует дефолт:
   SMECM, SMEAM, SMECIS, SMEND

3. Подключается к Jira REST API и выгружает по каждому проекту:
   - список задач за указанный период
   - changelog/history по задачам
   - sprint-данные, если доступны
   - поля:
     key, summary, issueType, status, priority, created, updated, resolved,
     assignee, reporter, labels, components, fixVersions,
     story points / customfield если доступно,
     sprint field / customfield если доступно,
     resolution,
     changelog histories

4. Приложение должно уметь работать с Jira 8.20.x:
   - использовать REST API, совместимый с Server/Data Center
   - аккуратно обрабатывать кастомные поля
   - пытаться автоматически определить поле Story Points и Sprint
   - если автоопределение не удалось — позволять указать ID полей в конфиге

5. Для Scrum-проектов посчитать немногочисленные, но полезные метрики по каждому проекту.

====================================
МЕТРИКИ
====================================

Нужны 2 уровня метрик:
1. CEO-level summary
2. Detailed report

Сделай метрики такими.

CEO-level:
- backlog_stability_index
  Доля задач, у которых priority не менялся после попадания в активную работу / sprint
- scope_change_rate
  Доля задач, добавленных в sprint после его старта
- completion_rate
  Завершенные задачи / взятые в работу задачи
- wip_overload_index
  Прокси перегрузки: среднее количество одновременно активных задач
- predictability_index
  Стабильность cycle time, например через coefficient of variation или другой понятный показатель

Detailed:
- total_issues_analyzed
- total_sprints_analyzed
- priority_changes_total
- issues_with_priority_changes_pct
- issues_with_priority_change_after_start_pct
- avg_priority_changes_per_issue
- tasks_added_after_sprint_start_pct
- tasks_removed_from_sprint_pct, если это можно извлечь
- avg_cycle_time_days
- median_cycle_time_days
- p85_cycle_time_days
- reopened_issues_pct
- cancelled_or_not_done_pct
- avg_wip
- max_wip

- aging_in_progress_over_14d
- backlog_items_without_estimate_pct
- backlog_items_without_description_pct

Если часть метрик нельзя надежно посчитать из стандартного API Jira 8.20.x без доп. плагинов, не ломай приложение:
- логируй предупреждение
- помечай метрику как unavailable
- продолжай работу

====================================
ЛОГИКА РАСЧЕТА
====================================

Используй прагматичную и прозрачную логику.

1. Активная работа:
Считать, что задача вошла в активную работу, если:
- изменила статус на один из in progress статусов
  или
- попала в активный sprint

Список in-progress статусов должен быть настраиваемым через конфиг.
По умолчанию включи:
- In Progress
- Selected for Development
- Code Review
- QA
- Testing

2. Priority churn:
Считать изменение приоритета проблемным, если оно произошло после начала активной работы.

3. Scope change:
Если задача была добавлена в sprint после даты старта sprint — это scope change.

4. Carryover:
Если задача была в sprint, но не была завершена и перешла в следующий sprint — учитывать как carryover.

5. Completion rate:
Считать по задачам, которые были начаты в анализируемом периоде.

6. WIP:
Считать по истории переходов в активные статусы и выхода из них.
Если точный посуточный расчет сложен, реализовать разумную аппроксимацию и явно описать это в коде и в отчете.

7. Predictability:
Основной показатель — coefficient of variation для cycle time.
Дополнительно можно вернуть интерпретацию:
- good
- medium
- poor
