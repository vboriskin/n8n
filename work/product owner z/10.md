# Jira Backlog Audit

Production-like CLI-инструмент для анализа качества backlog management в Jira Server / Data Center v8.20.11.

Инструмент позволяет:
- загружать issues из Jira по списку project keys;
- разбирать changelog;
- строить нормализованные таблицы;
- считать метрики качества backlog management;
- формировать summary, scorecard, flagged issues;
- строить отчеты и графики.

---

## 1. Что анализируется

Инструмент оценивает backlog management по пяти блокам:

1. **Priority stability**
   - доля задач со сменой priority;
   - доля задач, где priority менялся после начала работы;
   - интенсивность churn приоритетов.

2. **Unplanned work**
   - доля задач, быстро стартовавших после создания;
   - прокси для внеплановой работы.

3. **Overload**
   - long in progress ratio;
   - aging WIP proxy;
   - weekly trend по старту задач.

4. **Readiness**
   - старт без оценки;
   - возвраты из активной работы назад в backlog.

5. **Predictability**
   - lead time;
   - cycle time;
   - cancelled ratio;
   - done ratio.

---

## 2. Ограничения текущей версии

Текущая версия — рабочая baseline-реализация, но с ограничениями:

- WIP считается приближенно;
- sprint churn / carry-over / plan-vs-fact пока не реализованы;
- часть метрик зависит от корректного status mapping;
- метрика readiness по оценке зависит от правильно заданного `story_points_field`;
- HTML отчет не интерактивный.

## 8. CLI команды

### `fetch`

Скачивает issues из Jira, включая changelog, и сохраняет raw JSON.

### `analyze`

Нормализует raw JSON и считает метрики.

### `report`

Строит итоговые таблицы, markdown/html отчеты и графики.

### `run`

Выполняет весь pipeline целиком:
`fetch -> analyze -> report`

10. МИНИ-ПОЛИШ ДЛЯ graceful degradation
=======================================

Ниже предлагается небольшое улучшение в `cli.py`: если часть таблиц пустая, pipeline не должен падать.

ЗАМЕНИ в `_run_metrics` участок с `pd.concat(...)` на безопасный вариант:

def _safe_concat(frames):
non_empty = [df for df in frames if df is not None and not df.empty]
if not non_empty:
return pd.DataFrame()
return pd.concat(non_empty, ignore_index=True)

и далее:

```
metrics_by_project_df = _safe_concat(
    [
        priority_metrics_df,
        unplanned_metrics_df,
        overload_metrics_df,
        readiness_metrics_df,
        predictability_metrics_df,
    ]
)

metrics_by_week_df = _safe_concat(
    [
        overload_weekly_df,
        predictability_weekly_df,
    ]
)

flagged_issues_df = _safe_concat(
    [
        priority_flagged_df,
        unplanned_flagged_df,
        overload_flagged_df,
        readiness_flagged_df,
        predictability_flagged_df,
    ]
)

if not flagged_issues_df.empty:
    flagged_issues_df = flagged_issues_df.drop_duplicates()
```

Это сделает pipeline устойчивее, если какая-то часть данных недоступна.

12. ФИНАЛЬНЫЙ РЕЗУЛЬТАТ ПРОЕКТА
===============================

На текущем этапе реализован end-to-end baseline pipeline:

1. CLI принимает:

   * base_url,
   * token,
   * массив project keys,
   * период анализа,
   * out_dir.

2. Fetch layer:

   * выгружает issues,
   * использует changelog,
   * сохраняет raw JSON.

3. Transform layer:

   * строит issues table,
   * changelog table,
   * status history,
   * priority history,
   * timeline facts.

4. Metrics layer:

   * считает priority stability,
   * unplanned work,
   * overload,
   * readiness,
   * predictability,
   * scorecard,
   * flagged issues.

5. Reporting layer:

   * сохраняет summary.csv,
   * detailed_metrics.csv,
   * flagged_issues.csv,
   * report.md,
   * report.html,
   * charts/*.png.
