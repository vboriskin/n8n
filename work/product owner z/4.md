ШАГ 1. Архитектура и структура проекта

Ниже предлагаю архитектуру production-ready CLI-проекта для анализа backlog quality в Jira Server / Data Center v8.20.11.

Цели архитектуры:
- разделить загрузку данных, нормализацию, расчет метрик и отчетность;
- упростить поддержку кастомных полей Jira;
- обеспечить graceful degradation при неполных данных;
- сделать проект расширяемым для новых метрик и визуализаций.

==================================================
1. ВЫСОКОУРОВНЕВАЯ АРХИТЕКТУРА
==================================================

Проект делится на 6 слоев:

1. CLI layer
   Отвечает за команды fetch / analyze / report / run, аргументы, orchestration.

2. Config layer
   Загружает настройки из YAML/TOML + env vars + CLI overrides.
   Содержит:
   - base_url,
   - token,
   - projects,
   - date range,
   - field mappings,
   - status mappings,
   - metric thresholds,
   - score weights,
   - output settings.

3. Jira access layer
   Инкапсулирует REST API Jira:
   - auth,
   - pagination,
   - retries/backoff,
   - JQL generation,
   - raw issue fetch,
   - optional agile/sprint fetch.

4. Transformation layer
   Преобразует raw Jira JSON в нормализованные таблицы:
   - issues,
   - changelog events,
   - status history,
   - priority history,
   - sprint membership,
   - per-issue timeline facts.

5. Metrics layer
   Считает метрики по проектам / неделям / спринтам:
   - priority stability,
   - unplanned work,
   - overload,
   - readiness,
   - predictability,
   - scoring.

6. Reporting layer
   Строит итоговые артефакты:
   - csv/parquet/json,
   - markdown report,
   - html report,
   - charts png/html.

==================================================
2. ОСНОВНОЙ ПОТОК ДАННЫХ
==================================================

Команда run должна выполнять пайплайн:

1. CLI получает параметры запуска.
2. Config layer собирает итоговый runtime config.
3. Fetch layer ходит в Jira REST API и сохраняет raw JSON.
4. Transformation layer нормализует raw-данные в табличные наборы.
5. Metrics layer считает диагностические показатели.
6. Scoring layer формирует scorecard.
7. Reporting layer сохраняет отчеты и визуализации.

Поток артефактов:

Jira REST API
  -> raw JSON
  -> normalized tables
  -> metrics tables
  -> score tables
  -> report + charts

14. GRACEFUL DEGRADATION
==================================================

Архитектурно важно предусмотреть частичную недоступность данных.

Если недоступны sprint data:
- не считаем sprint churn / carry-over / plan-vs-fact,
- но остальной анализ продолжаем.

Если не задан story points field:
- readiness metric “start without estimate” либо пропускается,
- либо считается по alternative estimate field из config.

Если changelog неполный:
- логируем предупреждение,
- считаем только те метрики, для которых достаточно данных.

Если часть проектов недоступна по правам:
- пропускаем только эти проекты,
- весь pipeline не падает.
