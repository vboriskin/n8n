Да — вот скорректированный промпт под вашу среду.

Я сделал акцент на том, что:

* Jira развернута **в корпоративной среде**;
* авторизация идет через **Bearer token** с плейсхолдером **`<ТОКЕН>`**;
* на вход подается **массив кодов пространств / проектов Jira** вроде `["SMECM", "ABC", "XYZ"]`;
* на выходе нужны **отчет и визуализация**;
* инструмент должен быть запускаемым как реальный проект.

```text
Тебе нужно спроектировать и реализовать production-ready CLI-проект для анализа качества управления backlog в Jira Server / Data Center v8.20.11, развернутой в корпоративной среде.

Цель
Нужно по набору Jira projects / spaces оценить качество backlog management и выявить признаки:
1. нестабильности приоритетов,
2. высокой доли внеплановой работы,
3. перегрузки команды,
4. низкого качества подготовки задач,
5. низкой предсказуемости исполнения.

Контекст
- Jira: Server / Data Center v8.20.11.
- Jira развернута внутри корпоративной среды.
- Доступ к Jira должен выполняться через REST API.
- Аутентификация: Bearer Token.
- В коде и примерах используй плейсхолдер:
  Authorization: Bearer <ТОКЕН>
- Я потом сам подставлю реальный токен.
- На вход утилите должен передаваться массив кодов Jira-проектов / пространств, например:
  ["SMECM", "PAY", "CORE"]
- На выходе должны формироваться:
  1) отчет,
  2) визуализация.

Требования к реализации
- Язык: Python 3.11+.
- Это должен быть именно CLI-проект, а не ноутбук.
- Код должен быть модульным, понятным, расширяемым.
- Используй typing, dataclasses или pydantic, логирование, аккуратную обработку ошибок.
- Не хардкодь project keys, статусы, workflow, custom fields.
- Все важные параметры должны задаваться конфигом и/или CLI-аргументами.
- Нужно учитывать пагинацию Jira REST API.
- Нужно минимизировать количество запросов.
- Исторические изменения задачи нужно анализировать через changelog.
- Решение должно быть готово к запуску на реальных данных.
- Предусмотри, что часть полей может отсутствовать, быть кастомной или называться по-разному в разных проектах.

Нужно реализовать CLI со следующими командами

1. fetch
Назначение:
- скачать данные из Jira по переданным проектам за период;
- сохранить raw JSON локально;
- сохранить нормализованные данные в удобном формате.

Пример запуска:
python -m jira_backlog_audit fetch \
  --base-url https://jira.company.local \
  --token "<ТОКЕН>" \
  --projects SMECM PAY CORE \
  --start-date 2025-10-01 \
  --end-date 2026-03-31 \
  --out-dir ./output

Что должна делать команда:
- получить issues по каждому проекту;
- использовать JQL по проекту и периоду;
- выгрузить основные поля issue;
- выгрузить changelog;
- при наличии Scrum-данных дополнительно получить sprint-related данные, если это возможно через REST API;
- сохранить raw ответы Jira;
- нормализовать данные в таблицы.

2. analyze
Назначение:
- прочитать выгруженные данные;
- посчитать метрики качества backlog management;
- сформировать агрегаты по проектам и по времени.

Пример:
python -m jira_backlog_audit analyze --input-dir ./output --out-dir ./output

3. report
Назначение:
- построить итоговые артефакты:
  - summary report,
  - подробный markdown/html отчет,
  - визуализации,
  - csv/parquet/json с результатами.

Пример:
python -m jira_backlog_audit report --input-dir ./output --out-dir ./output

4. run
Назначение:
- выполнить fetch + analyze + report за один запуск.

Пример:
python -m jira_backlog_audit run \
  --base-url https://jira.company.local \
  --token "<ТОКЕН>" \
  --projects SMECM PAY CORE \
  --start-date 2025-10-01 \
  --end-date 2026-03-31 \
  --out-dir ./output

Входные параметры
Инструмент должен поддерживать:
- --base-url
- --token
- --projects SMECM PAY CORE
- --start-date
- --end-date
- --config
- --out-dir
- --log-level

Также допустим вариант запуска через config-файл, но массив проектов должен поддерживаться и через CLI.

Что считать “пространством”
Для этой реализации считай, что “пространство” = Jira project key.
То есть входной массив вроде ["SMECM", "PAY"] означает анализ по этим Jira projects.

Источники данных Jira
Нужно использовать REST API Jira Server/DC v8.20.11.
Используй как минимум:
- /rest/api/2/search
- /rest/api/2/issue/{key} при необходимости
- expand=changelog там, где это возможно
- Agile endpoints, если они доступны и реально нужны для sprint analysis

Нужно собирать:
- issue key
- project key
- issue type
- summary
- status
- priority
- resolution
- created
- updated
- assignee
- reporter
- labels
- components
- fixVersions
- epic / parent / links, если доступны
- story points / estimate, если поле задано конфигом
- sprint field, если есть
- changelog:
  - изменения status
  - изменения priority
  - изменения assignee
  - изменения sprint
  - изменения estimate / story points при возможности

Обязательно сделай отдельный слой маппинга полей
Потому что в корпоративной Jira:
- story points могут быть customfield_XXXXX,
- sprint может быть customfield_XXXXX,
- названия статусов могут отличаться,
- “активные статусы” и “backlog-статусы” могут быть разными.

Поэтому нужен config, где можно задать:
- active_statuses
- done_statuses
- backlog_statuses
- cancelled_resolutions
- custom fields:
  - sprint_field
  - story_points_field
  - epic_link_field
  - team_field
  - any other optional fields

Метрики, которые нужно посчитать

Блок 1. Стабильность приоритетов
Для каждого проекта и в целом:
- доля задач, у которых priority менялся хотя бы 1 раз;
- среднее число смен priority на задачу;
- доля задач, у которых priority менялся после входа в активную работу;
- число смен priority на 100 задач;
- список задач с наиболее частыми сменами priority.

Блок 2. Внеплановая работа / “вбросы”
- доля задач, созданных и начатых в короткий срок (порог задается конфигом, например 1–3 дня);
- доля задач, начатых без заметного нахождения в backlog;
- доля задач, у которых время от создания до старта меньше порога;
- если доступны sprint data:
  - доля задач, добавленных в sprint после его старта,
  - sprint churn.

Блок 3. Перегрузка команды
- средний WIP по проекту;
- максимальный WIP;
- aging WIP: число активных задач старше 7/14/30 дней;
- среднее время нахождения в активной работе;
- доля задач, находящихся в работе дольше порога;
- WIP trend по неделям.

Блок 4. Качество подготовки задач
Если нужные поля доступны:
- доля задач, стартовавших без оценки;
- доля задач, стартовавших без обязательных полей;
- доля задач, вернувшихся из активной работы назад в backlog / analysis;
- число значимых изменений после старта:
  - priority,
  - estimate,
  - sprint,
  - assignee,
  - optionally description-related heuristics, если это возможно без перегрузки решения.

Блок 5. Предсказуемость исполнения
- lead time;
- cycle time;
- carry-over rate между спринтами, если sprint data доступны;
- доля задач, отмененных / закрытых как Won’t Do / Duplicate / Cancelled;
- доля задач, зависших и не завершенных в разумный срок;
- plan vs fact по sprint, если доступно;
- trend завершения задач по неделям.

Нормализация и модель данных
Сделай нормализованную модель данных, например:
- issues.parquet
- issue_changelog.parquet
- issue_status_history.parquet
- issue_priority_history.parquet
- sprint_membership.parquet
- metrics_by_project.parquet

Нужна отдельная логика:
- для извлечения changelog;
- для построения timeline задачи;
- для определения first start date;
- для определения first done date;
- для определения факта возврата назад;
- для определения, менялся ли priority после начала работы;
- для реконструкции WIP по дням или по неделям.

Визуализация
На выходе обязательно должна быть визуализация.
Нужно построить как минимум:
1. bar chart:
   - доля задач со сменой priority по проектам;
2. bar chart:
   - доля внеплановой работы по проектам;
3. bar chart:
   - средний / максимальный WIP по проектам;
4. line chart:
   - WIP trend по неделям для каждого проекта;
5. line chart:
   - trend входящего потока задач и завершенных задач по неделям;
6. histogram или boxplot:
   - cycle time / lead time по проектам;
7. heatmap или table-like visualization:
   - итоговый scorecard по проектам.

Используй matplotlib и/или plotly.
Отчеты должны сохраняться в файлы.
Минимальный набор выходных артефактов:

- report.md

- при желании charts/*.html

Отчет
Сформируй понятный итоговый отчет:
1. Executive summary
   - краткие выводы по каждому проекту.
2. Сводная таблица
   - проект,
   - количество задач,
   - доля смен priority,
   - доля внеплановой работы,
   - средний WIP,
   - aging WIP,
   - возвраты назад,
   - доля отмен,
   - lead time / cycle time,
   - итоговый risk level.
3. Подробный разбор по каждому проекту
   - ключевые наблюдения,
   - аномалии,
   - список проблемных задач,
   - интерпретация метрик.
4. Recommendations
   - общие рекомендации;
   - рекомендации по конкретным проектам.

Скоринг
Сделай не только сырые метрики, но и нормализованный scorecard.
Например по каждому проекту:
- priority stability score: 0..100
- unplanned work score: 0..100
- overload score: 0..100
- readiness score: 0..100
- predictability score: 0..100
- total risk score или total health score

Важно:
- формулы должны быть прозрачными;
- веса должны быть вынесены в config;
- в отчете нужно показать не только score, но и underlying metrics.
```
