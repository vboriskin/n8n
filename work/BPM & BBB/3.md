# 🔷 Слайд 1: Почему не хореография

**Заголовок:**

> Хореография не обеспечивает управление бизнес-процессами

**Левая часть (как работает хореография):**

* Сервисы обмениваются событиями
* Логика процесса распределена
* Нет центрального управления

(можно визуально: много сервисов + стрелки событий между ними)

---

**Правая часть (проблемы):**

**❌ Нет runtime-контроля**

* нельзя остановить или изменить процесс
* нельзя вмешаться в конкретный кейс

**❌ Нет единого состояния**

* процесс “размазан” по сервисам
* сложная диагностика и разбор инцидентов

**❌ Изменения только через релизы**

* правки = изменения в коде
* долгий time-to-market

**❌ Не заменяет BPM (Pega)**

* нет управления кейсами
* нет lifecycle-контроля

---

**Финальный вывод (жирно):**

> Хореография подходит для интеграций,
> но не решает задачу управления бизнес-процессами

# 🔷 Слайд 2: Почему orchestration

**Заголовок:**

> Orchestration layer как замена BPM (Pega)

---

**Левая часть (что было нужно сохранить):**

После отказа от Pega нам критично:

* управление процессами
* контроль выполнения (runtime)
* работа с кейсами
* наблюдаемость и диагностика

---

**Правая часть (что дает наше решение):**

**✔ Централизованное управление**

* единая точка контроля процессов

**✔ Runtime-изменения**

* изменения без релизов
* адаптация под регулятора

**✔ Управление кейсами**

* остановка / перезапуск / изменение
* работа с “зависшими” процессами

**✔ Наблюдаемость**

* единое состояние процесса
* прозрачный lifecycle

**✔ Масштабируемость**

* orchestration layer ≠ монолит
* горизонтальное масштабирование

---

**Финальный вывод (жирно):**

> Orchestration — единственный подход,
> который сохраняет управляемость процессов после отказа от Pega

# 🔷 Диаграмма (для устного объяснения или отдельного слайда)

## Вариант 1 — “Почему ломается хореография”

```
[Service A] → event → [Service B] → event → [Service C]
       ↘                    ↘
        event               event
         ↘                  ↘
        [Service D]      [Service E]
```

Подпись:

> ❗ Где находится процесс?
> ❗ Кто им управляет?
> ❗ Как его остановить?

Ответ: **нигде и никто**

---

## Вариант 2 — Orchestration

```
            [ Orchestration Layer ]
                     |
     ---------------------------------------
     |           |           |             |
 [Service A] [Service B] [Service C] [Service D]
```

Подпись:

> ✔ Процесс централизован
> ✔ Есть контроль и состояние
> ✔ Можно управлять в runtime

# 🔥 Короткий спич (очень рекомендую выучить)

> «После отказа от Pega нам нужно было сохранить не технологию, а capability — управление процессами.
>
> Хореография не дает runtime-контроля: процесс распределен, им нельзя управлять, нельзя вмешиваться в кейсы.
>
> Поэтому мы выбрали orchestration layer — как единственный подход, который позволяет централизованно управлять процессами, изменять их без релизов и обеспечивать наблюдаемость.»
