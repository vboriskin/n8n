Да. Ниже даю тебе **целый пакет безопасных LLM prompts** под 3 режима:

* `matrix_only`
* `mixed`
* `full`

Цель одна: **не галлюцинировать, не путать отсутствие данных с низким уровнем, не делать ложных performance-выводов**.

Я дам:

1. **общие принципы**
2. **единые mode rules**
3. prompts для:

   * competency analysis
   * employee summary
   * jira summary
   * IDP
   * one2one employee
   * one2one manager
   * development strategy
   * promotion readiness
   * external learning suggestions
4. **рекомендации по JSON schema enforcement**
5. **какие ноды к каким prompt’ам привязать**

---

# 1. Главный принцип всей системы

LLM у тебя должен делать только 3 вещи:

* **интерпретировать**
* **суммировать**
* **формулировать development guidance**

LLM **не должен**:

* придумывать evidence
* трактовать matrix как factual performance proof
* ставить высокий confidence при слабых данных
* считать coverage / gaps / scores, если это уже посчитано кодом

# 2. Единые правила для всех prompts

Вот этот блок я советую почти везде держать одинаковым.

## Universal System Safety Block

```text
Ты анализируешь данные сотрудника.

Жесткие правила:
- Используй только переданные данные
- Не выдумывай факты, evidence, достижения, активности, проекты, влияние, лидерство или результаты
- Если данных недостаточно, скажи это явно
- Не путай "нет данных" и "низкий уровень"
- Не путай role expectations из matrix и фактическое performance evidence
- Если factual evidence отсутствует, confidence должен быть low
- Если есть только matrix/rubric данные, фокусируйся на development guidance, expectations, learning plan и question prompts
- Если есть mixed data, отделяй factual evidence от role expectations
- Если есть противоречие между self-assessment и evidence, отмечай это аккуратно и без категоричности
- Отвечай строго в JSON
- Не добавляй markdown
- Не добавляй комментарии вне JSON
```

# 3. Mode rules — универсальный блок, который надо подмешивать в prompts

Это очень важно. Лучше явно прокидывать `assessment_mode` в payload и заставлять модель соблюдать режим.

## Mode rules text

```text
Правила интерпретации по режиму:

1) Если assessment_mode = "matrix_only":
- matrix описывает role expectations, а не фактический performance сотрудника
- не делай сильных выводов о реальном уровне по работе
- не используй language вроде "сотрудник слабый" или "сотрудник не умеет", если нет factual evidence
- фокусируйся на:
  - зонах развития
  - ожидаемых проявлениях компетенции
  - возможных шагах развития
  - вопросах для уточнения
- confidence должен быть low, если нет factual evidence

2) Если assessment_mode = "matrix_plus_issues" или "mixed":
- отличай factual evidence от matrix expectations
- factual evidence может поддерживать выводы, но только в пределах доступных данных
- если evidence неполное, не делай чрезмерно уверенных выводов
- matrix используй как baseline expectation, а не как proof

3) Если assessment_mode = "full":
- можно делать более содержательные выводы, но только на основе фактов из данных
- даже в full режиме не выдумывай evidence
- если evidence слабое по конкретной компетенции, confidence по этой компетенции должен быть ниже
```

# 4. Prompt 1 — Competency Analysis

Это самый важный prompt.

## System

```text
Ты анализируешь одну компетенцию сотрудника.

Жесткие правила:
- Используй только переданные данные
- Не выдумывай factual evidence
- Не путай отсутствие данных с низким уровнем
- Не путай matrix expectations и фактическое performance evidence
- Если self-assessment противоречит evidence, отмечай это аккуратно
- Если factual evidence нет, confidence должен быть low
- Если evidence только synthetic/matrix-based, не делай сильный performance verdict

Правила интерпретации по режиму:

1) Если assessment_mode = "matrix_only":
- matrix описывает role expectations, а не фактический performance сотрудника
- не делай уверенный вывод о реальном уровне сотрудника
- фокусируйся на development guidance и missing evidence
- estimated_level не должен выглядеть как уверенная оценка фактического уровня при отсутствии evidence

2) Если assessment_mode = "matrix_plus_issues" или "mixed":
- используй factual evidence осторожно
- matrix используй как baseline expectation
- отделяй observed signals от expected behaviors

3) Если assessment_mode = "full":
- используй factual evidence как основной источник
- confidence зависит от полноты и качества evidence

Верни строго JSON без markdown.
```

## User

```text
Верни JSON по схеме:
{
  "competency_id": "string",
  "self_level": 0,
  "rule_based_level": 0,
  "estimated_level": 0,
  "confidence": "low|medium|high",
  "assessment": "string",
  "evidence_for": ["string"],
  "evidence_missing": ["string"],
  "risk_flags": ["string"],
  "priority": "low|medium|high",
  "development_focus": "string"
}

Дополнительные правила:
- Если self_level отсутствует, верни 0
- Если rule_based_level отсутствует, верни 0
- Если evidence недостаточно, estimated_level может быть 0 или очень консервативным
- evidence_for должен содержать только то, что действительно есть во входных данных
- evidence_missing должен перечислять, каких фактов не хватает для более уверенного вывода
- assessment должен быть коротким, фактическим и без выдуманных деталей
- development_focus должен быть практичным и конкретным

Данные:
{{ JSON.stringify($json.payload) }}
```

# 5. Prompt 2 — Employee Summary

Нужен безопасный summary, чтобы не было ложных выводов в `matrix_only`.

## System

```text
Ты строишь общий summary по сотруднику на основе результатов анализа компетенций.

Жесткие правила:
- Используй только переданные данные
- Не выдумывай достижения, влияние, лидерство или delivery
- Не путай отсутствие данных с отсутствием компетенции
- Не путай matrix expectations с фактической оценкой performance
- Если confidence низкий, отмечай это явно
- Если assessment_mode = matrix_only, summary должен быть development-oriented, а не evaluative

Правила интерпретации по режиму:

1) matrix_only:
- не делай жестких выводов о реальной эффективности сотрудника
- фокусируйся на:
  - направлениях развития
  - зонах, где нужны дополнительные данные
  - приоритетах развития
- strengths в этом режиме можно формулировать только как:
  - areas with better alignment
  - areas with stronger baseline fit
  - areas with clearer expectations
  но не как доказанные рабочие достижения

2) mixed:
- различай observed evidence и matrix baseline
- summary должен быть осторожным

3) full:
- можно давать более содержательный общий summary, но строго по evidence

Верни строго JSON без markdown.
```

## User

```text
Верни JSON по схеме:
{
  "overall_summary": "string",
  "strengths": ["string"],
  "growth_areas": ["string"],
  "alignment": {
    "overestimated": ["string"],
    "underestimated": ["string"],
    "aligned": ["string"]
  },
  "top_priorities_next_6m": ["string"],
  "confidence": "low|medium|high"
}

Дополнительные правила:
- В matrix_only режиме overall_summary должен явно показывать, что выводы ограничены доступными данными
- strengths и growth_areas должны опираться только на входные данные
- top_priorities_next_6m должны быть конкретными competency_id или явно совпадающими с данными компетенций
- confidence должен отражать полноту данных, а не “общее впечатление”

Данные:
{{ JSON.stringify($json) }}
```

# 6. Prompt 3 — Jira Summary

Этот prompt вообще не надо запускать в `matrix_only`.
Но если все-таки одна ветка на все режимы, то надо его застраховать.

## System

```text
Ты анализируешь активность сотрудника по данным Jira.

Жесткие правила:
- Используй только переданные Jira/issue/activity данные
- Не выдумывай активность
- Worklog означает участие, но сам по себе не доказывает высокий уровень
- Commits, pull requests и merged pull requests — это инженерный сигнал, но не автоматическое доказательство seniority
- Если Jira/activity данных мало или нет, укажи это прямо
- Не путай отсутствие Jira данных с низкой активностью

Если assessment_mode = "matrix_only":
- не делай Jira-based performance summary
- верни безопасный ограниченный ответ с low confidence

Верни строго JSON без markdown.
```

## User

```text
Верни JSON по схеме:
{
  "overall_summary": "string",
  "activity_level": "low|medium|high",
  "delivery_assessment": "string",
  "engineering_signal": "string",
  "risks": ["string"],
  "confidence": "low|medium|high"
}

Дополнительные правила:
- Если Jira data отсутствует, overall_summary должен прямо это сказать
- Если evidence мало, confidence = low
- Не заполняй delivery_assessment или engineering_signal выдуманными фактами
```

# 7. Prompt 4 — IDP

Это очень важный prompt, особенно в `matrix_only`.

## System

```text
Ты создаешь индивидуальный план развития сотрудника.

Жесткие правила:
- Используй только переданные данные
- Не выдумывай фактические слабые или сильные стороны без evidence
- Если assessment_mode = matrix_only, ориентируйся на matrix expectations, learning opportunities и development guidance, а не на performance judgment
- План должен быть конкретным, реалистичным и проверяемым
- Используй mix из:
  - project work
  - learning
  - mentoring
  - reading
  - practice/review activities
- Если есть internal learning assets, используй их приоритетно
- Не ограничивай развитие только курсами

Правила по режиму:
- matrix_only: development based on role expectations and missing evidence
- mixed: combine observed data with development opportunities
- full: use competency results and evidence more confidently

Верни строго JSON без markdown.
```

## User

```text
Верни JSON по схеме:
{
  "idp": [
    {
      "competency_id": "string",
      "goal": "string",
      "priority": "low|medium|high",
      "actions": [
        {
          "type": "task|learning|mentoring|project|reading",
          "description": "string",
          "effort": "low|medium|high"
        }
      ],
      "success_criteria": ["string"],
      "timeframe": "string"
    }
  ]
}

Дополнительные правила:
- goal должен быть кратким и прикладным
- actions должны быть конкретными и выполнимыми
- success_criteria должны быть проверяемыми
- не используй абстрактные формулировки вроде "стать лучше"
- если у компетенции мало factual evidence, можно строить goal как развитие или валидацию компетенции, а не исправление "слабости"

Данные:
{{ JSON.stringify($json) }}
```

# 13. Как подключать эти prompts по режимам

Я бы сделал так.

## Для `matrix_only`

Использовать:

* Competency Analysis — безопасная версия
* Employee Summary — безопасная version
* IDP
* One2One Employee
* One2One Manager
* Development Strategy
* External Learning
* Promotion Readiness — только если очень нужен, и с консервативным output

### Не использовать

* Jira Summary как основной смысловой output

---

## Для `mixed`

Использовать все:

* Competency Analysis
* Employee Summary
* IDP
* One2One Employee
* One2One Manager
* Development Strategy
* External Learning
* Jira Summary
* Promotion Readiness

Но prompts должны сохранять distinction:

* factual evidence
* matrix baseline

---

## Для `full`

Те же prompts, но можно:

* не менять тексты
* просто дать богатый payload
* и mode rules сами разрешат более уверенные выводы

# 14. Очень важная штука: schema-first parsing

Чтобы LLM меньше “плыл”, я очень советую:

* давать **строгую схему JSON**
* не просить “расскажи подробно”
* не просить “объясни свои рассуждения”
* не просить несколько альтернатив

Идеально:

* 1 задача
* 1 JSON
* 1 понятная структура

# 15. Дополнительные безопасные post-processing rules в Code nodes

Даже с хорошими prompts лучше подстраховаться кодом.

## После Competency Analysis

Если:

* `assessment_mode = matrix_only`
* и `evidence_for` не содержит factual evidence
  то:
* принудительно `confidence = low`

## После Promotion Readiness

Если:

* `assessment_mode = matrix_only`
  то:
* если readiness = `ready`, понижаем до `emerging`
  или даже `not_ready`, если совсем нет evidence

## После Employee Summary

Если:

* `assessment_mode = matrix_only`
* и summary звучит слишком уверенно
  можно regex-проверкой убирать формулировки типа:
* “доказал”
* “демонстрирует высокий уровень”
* “стабильно показывает высокий результат”

# 17. Мой практический совет

Если хочешь минимизировать галлюцинации сильнее всего, начни с такого порядка внедрения:

1. **Competency Analysis**
2. **Employee Summary**
3. **IDP**
4. **One2One Employee**
5. **One2One Manager**

А уже потом:

* Promotion Readiness
* External Learning
* Development Strategy
* Jira Summary

Потому что именно первые 5 дают основной value и при этом легче контролируются.

Следующим сообщением я могу собрать тебе это уже в **n8n-ready формате**: по каждой ноде отдельно, с готовыми `System` и `User` блоками без лишней теории, чтобы ты просто копипастил.
