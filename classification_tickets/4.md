Да — текущий вариант рабочий, но его можно заметно улучшить по трём направлениям:

1. **качество классификации**
2. **скорость и цена**
3. **управленческая полезность результата**

Ниже дам именно инженерные улучшения для n8n + OpenAI.

---

# Куда реально улучшать

Сейчас у тебя базовая схема:

`xlsx -> clean -> batch -> OpenAI -> parse -> save`

Она нормальна для MVP, но на 39k строк быстро всплывут проблемы:

* дорого
* долго
* нестабильный JSON на длинных батчах
* разные формулировки root cause
* много “пустых” тикетов, на которые LLM тратится зря
* сложно отличить массовые шаблонные кейсы от реально уникальных проблем

Поэтому я бы улучшал так.

---

# 1. Сделать не один LLM-шаг, а каскад

Вместо одного жирного вызова модели лучше сделать **маршрутизацию**.

## Новый pipeline

### Этап A. Правила без LLM

Сначала простая дешёвая логика:

* если `ТЕГ`, `Тема`, `Код закрытия`, `Приоритет`, `Возвраты` уже однозначно говорят о типе тикета — не слать в LLM
* если это явно доступы / консультация / типовая ручная операция — классифицировать правилами
* если текст слишком короткий или мусорный — маркировать `low_info`

### Этап B. LLM только для сложных кейсов

В OpenAI отправляются:

* неоднозначные тикеты
* длинные описания
* случаи, где надо понять root cause
* случаи, где надо определить сторону ответственности

### Этап C. Пост-нормализация

После LLM:

* схлопнуть похожие причины
* схлопнуть похожие темы
* досчитать risk score

Это даст большой выигрыш по цене и скорости.

---

# 2. Добавить rule-based preclassifier

Это один из самых полезных апгрейдов.

## Зачем

У тебя из 39k тикетов часть наверняка:

* типовые доступы
* консультации
* запросы на ручную корректировку
* маршрутизация
* явные дубли

Для них LLM избыточна.

## Что можно определять правилами

Через `Code` node:

* `request_type_pre`
* `root_cause_group_pre`
* `is_simple_case`
* `needs_llm`

Например:

* если в тексте есть `доступ`, `роль`, `право`, `разблокировать` → `access`
* если есть `консультация`, `разъяснить`, `пояснить` → `consultation`
* если есть `исправить данные`, `скорректировать`, `обновить атрибут` → `data_fix`
* если есть `не работает`, `ошибка`, `сбой`, `некорректно`, `не отображается` → `incident_candidate`
* если `rework_count > 0` или `overdue_flag=yes` → флаг повышенного внимания

## Что это даст

Иногда 30–60% потока можно не гонять через модель вообще.

---

# 3. Сделать двухуровневую модель: mini + full

Если в n8n есть доступ к нескольким моделям, лучше не всё слать в одну.

## Подход

### Модель 1 — дешевая/быстрая

Для:

* summary
* request_type
* простого root_cause_group
* triage

### Модель 2 — сильнее, но только на сложные

Для:

* fault attribution
* нормализованного root cause
* automation candidate
* ambiguous cases

## Практический эффект

* быстрее
* дешевле
* меньше нагрузка на лимиты
* сильную модель тратишь только там, где она реально нужна

---

# 4. Ввести confidence gating

Сейчас `confidence` просто возвращается моделью.
Лучше использовать его как управляющий механизм.

## Логика

Если:

* `confidence >= 0.8` → принять результат
* `0.5 <= confidence < 0.8` → отправить на второй проход упрощённым prompt
* `< 0.5` → пометить как `manual_review`

## Что это даст

Ты не будешь строить выводы на шаткой классификации.

---

# 5. Добавить “обязательные сигналы” до LLM

Перед моделью полезно вычислить инженерные признаки.

## Фичи, которые стоит добавить в pre-processing

### текстовые

* длина описания
* длина решения
* количество слов
* число восклицаний / caps / технических кодов
* наличие ключевых паттернов: ошибка, доступ, корректировка, не работает, вручную, повторно, вернуть, согласовать, миграция

### операционные

* `has_rework`
* `is_overdue`
* `duration_bucket`
* `priority_bucket`
* `is_closed_successfully`
* `executor_group_changed` если есть такие данные
* `initiator_department_risk_segment`

### смысловые

* `text_signal_score`
* `system_issue_candidate`
* `human_error_candidate`
* `access_candidate`
* `data_fix_candidate`

Эти признаки можно прямо подмешивать в prompt.
Модель на таком входе будет заметно стабильнее.

---

# 6. Не слать целый текст, а делать smart compression

На длинных тикетах модель будет дорогой и нестабильной.
Лучше делать не просто `truncate`, а **выжимку**.

## Вариант лучше текущего

Перед основной классификацией сделать легкий summarization step:

### Шаг 1

Малая модель извлекает:

* суть запроса
* что сломалось
* что сделали
* были ли признаки ошибки пользователя
* были ли признаки системного дефекта

### Шаг 2

Основная модель получает:

* структурированные поля
* короткую выжимку вместо всей простыни текста

Это сильно режет токены.

---

# 7. Добавить дедупликацию до LLM

Очень вероятно, что среди 39k тикетов много почти одинаковых обращений.

## Как сделать без embeddings

Простой fingerprint:

* `normalized(topic + tag + first_300(description) + executor_group + close_code)`

Потом:

* убрать стоп-слова
* lowercase
* выкинуть числа/ID
* схлопнуть пробелы
* hash

Если одинаковые fingerprint’ы повторяются часто:

* прогоняешь один representative ticket через LLM
* остальным наследуешь классификацию

## Что даёт

На реальных сервисных данных может сэкономить очень много запросов.

---

# 8. А ещё лучше — кластеризация похожих кейсов

Если хочешь следующий уровень зрелости:

## Сначала

собираешь для каждого тикета:

* topic
* tag
* краткий текст
* resolution

## Потом

строишь embedding и ищешь похожие тикеты.

## Что получаешь

* recurring issues
* топ массовых шаблонов
* кандидаты на автоматизацию
* не 39k отдельных записей, а 300–800 кластеров

И вот уже кластеры можно анализировать управленчески.

Это один из самых сильных апгрейдов всей системы.

---

# 9. Делать LLM не на тикет, а на кластер

Это вообще может стать главным ускорением.

## Схема

1. Правила + дедуп + rough clustering
2. Для каждого кластера:

   * cluster summary
   * canonical root cause
   * cluster automation potential
3. Потом обратно мапишь на тикеты

## Плюс

Не 39 000 LLM-решений, а, например, 1 500–3 000.

---

# 10. Разделить “что случилось” и “кто виноват”

Сейчас ты хочешь и тему, и причину, и ответственность в одном вызове.
Но “вина команды” — самое шаткое поле.

Лучше развести.

## Шаг 1 — объективная классификация

* тип обращения
* проблема
* root cause
* resolution type

## Шаг 2 — attribution layer

Отдельная логика:

* правила + LLM
* вина команды только если есть сигналы

## Почему это лучше

Модель не будет подгонять root cause под поиск виноватого.

---

# 11. Сделать fault attribution более строгим

Чтобы не получить токсичный отчёт “вот эти косячат”, нужно ввести уровни.

## Вместо одного поля

Добавь:

* `fault_signal_strength`: low / medium / high
* `fault_evidence`: текстовое краткое объяснение
* `fault_attribution_basis`: explicit / inferred / weak_signal
* `blame_safe_flag`: yes / no

## Польза

В отчёте можно показывать:

* только high-confidence attribution
* а всё остальное — как “зона риска”, а не “вина”

Это сильно повышает доверие к аналитике.

---

# 12. Добавить explainability поля

Очень рекомендую. Иначе потом никто не поймёт, почему тикет попал в категорию.

## Пусть модель возвращает ещё

* `decision_rationale_short`
* `evidence_snippets`
* `key_signals`

Например:

* `key_signals`: ["упоминается ручная корректировка", "есть возврат", "признаков системного сбоя нет"]

Можно ограничить это 1–2 короткими фразами.

## Это полезно для

* ручной валидации
* защиты результата перед заказчиком
* дообучения правил

---

# 13. Добавить нормализацию оргструктуры

Почти всегда в таких выгрузках бардак в названиях:

* команда
* рабочая группа
* исполнитель
* инициаторское подразделение

## Что сделать

Перед аналитикой ввести справочники:

* canonical team
* canonical executor group
* canonical initiator department

Например:

* `Поддержка Hydra`
* `РГ Гидра`
* `Hydra support`
* `Hydra`

должны схлопываться в одну сущность.

Иначе у тебя аналитика по командам поплывёт.

---

# 14. Добавить нормализацию тегов

ТЕГ почти наверняка хаотичный.

## Что можно сделать

Разбить `#...#`, очистить и привести к канону:

* `It проблема`
* `IT проблема`
* `айти проблема`

→ `it_problem`

* `перемиграция`
* `запрос перемиграции`

→ `migration_request`

Это можно сделать без LLM на словаре и regex.

---

# 15. Считать не только тему и причину, но ещё тип ценности

Очень полезно для управленческого решения.

## Добавь поля

* `is_failure_demand` — тикет возник из-за сбоя/дефекта
* `is_value_demand` — это нормальный сервисный запрос
* `is_avoidable` — можно ли было избежать тикета
* `is_repeatable_pattern` — массовый шаблон
* `is_automation_candidate`

Это потом отлично работает в отчётах:

* сколько потока вообще “лишнее”
* сколько нагрузки создано дефектами процесса
* сколько можно снять автоматизацией

---

# 16. Добавить бизнес-важность тикета

Приоритет не всегда отражает реальное влияние.

Можно LLM или правилами оценивать:

* `business_impact`: low / medium / high / critical
* `customer_impact`: none / limited / significant
* `operational_risk`: low / medium / high

Тогда ты увидишь не просто частотность, а **что болит сильнее всего**.

---

# 17. Сделать отдельный слой “automation opportunities”

Сейчас это одно поле `automation_candidate`. Этого мало.

## Лучше добавить

* `automation_candidate`
* `automation_type`
* `automation_target_step`
* `automation_complexity`
* `expected_benefit`

Например:

* `automation_type`: routing / data_check / notification / data_fix / decision_support
* `automation_target_step`: валидация входных данных / ручное исправление / маршрутизация / сверка статусов
* `automation_complexity`: low / medium / high
* `expected_benefit`: low / medium / high

Тогда ты сможешь не просто сказать “можно автоматизировать”, а построить roadmap.

---

# 18. Добавить anomaly detection

Это уже продвинутый, но очень полезный слой.

Ищи:

* редкие причины с большой длительностью
* необычные сочетания `team + topic + root_cause`
* исполнителей с аномально высоким rework
* подразделения инициаторов с аномально высоким user_error

Даже без LLM это даёт много пользы.

---

# 19. Разнести обработку на async очереди внутри n8n

Чтобы ускорить выполнение.

## Сейчас

Если workflow идёт последовательно батч за батчем — будет долго.

## Лучше

Разбить на:

* ingestion workflow
* classification worker workflow
* aggregation workflow

То есть:

1. первый workflow кладёт строки/батчи в таблицу `tickets_queue`
2. второй workflow по cron/trigger берёт пачку необработанных
3. пишет результат в `tickets_enriched`
4. третий строит сводки

Это гораздо устойчивее, чем один длинный прогон.

---

# 20. Перестать хранить всё только в CSV

Для 39k строк MVP на CSV возможен, но потом станет неудобно.

Лучше сразу:

* PostgreSQL / MySQL / SQLite хотя бы
* отдельные таблицы:

  * `tickets_raw`
  * `tickets_preprocessed`
  * `tickets_enriched`
  * `classification_errors`
  * `canonical_mappings`

## Почему

* можно дозапускать только хвост
* можно повторно переразметить часть
* проще контролировать ошибки
* проще строить BI

---

# 21. Сделать инкрементальную обработку

Не перегонять весь файл каждый раз.

## Добавь

* hash по ключевым полям тикета
* `processed_at`
* `model_version`
* `prompt_version`

Если тикет уже обработан и hash не изменился — пропускаешь.

Это критично, если выгрузка будет регулярной.

---

# 22. Версионировать prompt и модель

Очень советую.

Сохраняй в каждую запись:

* `prompt_version`
* `model_name`
* `workflow_version`
* `preprocess_version`

Потом сможешь сравнивать:

* почему старые и новые результаты отличаются
* стало ли лучше после изменения prompt

---

# 23. Добавить human-in-the-loop слой

На 39k строк полностью без ручной проверки лучше не идти.

## Что сделать

Выборка на ревью:

* все `confidence < 0.6`
* все `team_fault_flag = yes`
* все `responsible_side = executor_team`
* все `automation_candidate = yes` с `expected_benefit = high`
* все редкие кластеры

Это позволит быстро калибровать систему.

---

# 24. Ввести gold sample для проверки качества

Сделай руками 200–300 размеченных тикетов.

По ним сравнивай:

* совпадение `request_type`
* совпадение `root_cause_group`
* precision для `team_fault_flag=yes`
* agreement по `responsible_side`

Без этого система будет “выглядеть умной”, но качество не измеришь.

---

# 25. Улучшить prompt: запрет на обвинение без признаков

Это прямо важно.

Добавь правило:

> Не ставь `team_fault_flag = yes`, если нет явных или сильных косвенных признаков ошибки исполняющей стороны. В спорных случаях выбирай `unknown` или `partial`.

Это резко снизит ложные обвинения.

---

# 26. Улучшить prompt: разделить root cause и trigger

Полезное добавление:

* `trigger_event` — что стало непосредственным поводом обращения
* `root_cause` — глубинная причина

Например:

* trigger: “не отображается статус”
* root cause: “сбой интеграции”

Это делает аналитику намного взрослее.

---

# 27. Добавить процессные поля

Кроме root cause стоит вернуть:

* `process_stage`
* `failure_point`
* `handoff_issue_flag`

Например:

* процессный этап: регистрация / обработка / согласование / исполнение / закрытие
* точка сбоя: валидация данных / интеграция / маршрутизация / ручная операция / коммуникация

Это особенно полезно, если цель — не просто рейтинг команд, а улучшение процесса.

---

# 28. Сделать тематическую иерархию

Вместо одного `problem_theme` лучше две ступени:

* `theme_l1` — крупная категория
* `theme_l2` — уточнение

Например:

* `theme_l1 = кредитные операции`
* `theme_l2 = выдача кредитных средств`

И для причин:

* `cause_l1 = process`
* `cause_l2 = ручная корректировка данных`

Так сводки становятся намного удобнее.

---

# 29. Добавить multi-label, а не только single-label

Некоторые тикеты реально одновременно:

* data_fix
* process_issue
* automation_candidate

Не всё стоит сжимать в один класс.

Можно вернуть:

* основной тип
* вторичные признаки

Например:

* `primary_request_type`
* `secondary_labels`: ["data_fix", "manual_operation", "repeatable"]

Это очень полезно для аналитики улучшений.

---

# 30. Самое важное ускорение: сократить токены

Вот что даст максимальный практический эффект:

### сильно ускоряет и удешевляет:

* rule-based preclassification
* дедупликация
* кластеризация похожих тикетов
* smart summary перед main classification
* LLM only for ambiguous cases
* mini-model first, strong-model second
* инкрементальная обработка

Если делать только одно улучшение — я бы начал именно с этого набора.
