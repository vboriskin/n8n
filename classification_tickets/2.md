# Какой должен быть prompt

## Системный prompt для OpenAI-ноды

В n8n в OpenAI node лучше просить **строго JSON**.

Пример системного промпта:

```text
Ты аналитик сервиса Service Manager. 
Тебе передаются тикеты. 
Нужно классифицировать каждый тикет для управленческой аналитики.

Правила:
1. Используй только информацию из переданных полей.
2. Не выдумывай факты.
3. Если недостаточно данных, ставь unknown.
4. Определи вероятную корневую причину и сторону ответственности.
5. Ответ верни только в JSON-массиве без пояснений.
6. Для каждого тикета обязательно сохрани исходный id.
7. problem_theme и root_cause формулируй кратко и нормализованно.
8. suspected_fault_team указывай только если это реально следует из данных, иначе unknown.
9. team_fault_flag:
   - yes, если явно видно, что ошибка/недоработка на стороне исполнителя или команды
   - no, если проблема на стороне инициатора или это стандартный запрос
   - partial, если есть смешанная ответственность
   - unknown, если недостаточно данных
10. automation_candidate = yes, если это повторяющийся шаблонный ручной кейс, который можно автоматизировать.
```

# Как выявлять “косячные” команды корректно

Это надо делать не одним полем, а индексом.

Потому что “кто косячит” — опасная формулировка, если считать тупо по числу тикетов.
У команды может быть просто больше потока.

Лучше считать **несколько метрик**:

### по команде / рабочей группе

* `tickets_total`
* `tickets_with_team_fault`
* `% team_fault`
* `avg_duration`
* `median_duration`
* `% with_rework`
* `% overdue`
* `% system_defect`
* `% process_issue`
* `% automation_candidate`

---

## Нормальный score команды

Например:

```text
team_problem_score =
0.35 * fault_rate +
0.20 * rework_rate +
0.20 * overdue_rate +
0.15 * normalized_duration +
0.10 * defect_share
```

Тогда ты получишь не просто “много тикетов”, а **вероятность системной проблемности команды**.

# Что важно для стабильности

## 1. Делать retry

Если OpenAI вернул невалидный JSON:

* 1 retry с более жёстким prompt
* потом писать в error table

## 2. Не тащить лишние поля в prompt

Чем чище вход, тем лучше результат.

## 3. Ограничивать длину текста

Если описание огромное:

* брать первые N символов
* и последние N символов
  Часто суть в начале, решение — в конце.

Например:

* `description_first_2000`
* `resolution_first_1500`
