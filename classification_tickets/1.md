Задача у тебя классическая для service analytics: много “грязного” текстово-операционного лога + куча полей → нужно вытащить **структуру проблем + ответственность**. Ниже дам не теорию, а рабочие подходы от “быстро и просто” до “нормальная аналитическая система”.

---

# 🚀 1. Базовая стратегия (правильный порядок действий)

Раздели задачу на 3 слоя:

### 1) Нормализация и фичи

* почистить данные
* извлечь признаки

### 2) Классификация проблем

* темы / причины / типы

### 3) Аналитика ответственности

* где узкие места
* кто “косячит”
* где системные проблемы

---

# 🧠 2. Что реально можно вытащить из твоих данных

У тебя уже есть золото:

### 📊 Структурные поля:

* Команда
* Рабочая группа исполнителя
* Исполнитель
* Приоритет
* Код закрытия
* Длительность / SLA
* Возвраты

👉 Это даст:

* SLA-аналитику
* перформанс команд
* проблемные зоны

---

### 📝 Текст:

* ТЕГ (частично структурирован)
* Тема ЗНО
* Подробное описание (ограничено, но может быть полезно)
* Решение

👉 Это нужно для:

* кластеризации
* поиска реальных причин

---

# ⚙️ 3. Варианты решений

## 🟢 Вариант 1 — Быстрый (без LLM)

### Что делаем:

1. **Нормализуем ТЕГ**

   * разбиваем `#...#`
   * приводим к словарю

2. **Группируем Тема ЗНО**

   * они уже почти категории

3. Считаем:

* количество тикетов
* среднюю длительность
* % возвратов
* % просрочек

👉 Получишь:

* топ проблем
* топ плохих команд

📌 Минус: не увидишь “скрытые” причины

---

## 🟡 Вариант 2 — Кластеризация (без LLM)

Используем ML:

### Pipeline:

1. TF-IDF / embeddings (например:

   * `sentence-transformers`)
2. Кластеризация:

   * KMeans / HDBSCAN

### На вход:

* ТЕГ + Тема ЗНО + Решение

### На выход:

* группы типа:

  * “доступы”
  * “ошибки системы”
  * “миграции”
  * “ручные операции”

👉 Плюс:

* увидишь реальные темы, а не формальные

---

## 🔴 Вариант 3 — LLM (рекомендую)

Это самый мощный вариант для твоей задачи.

---

# 🤖 4. Как использовать LLM

## Задача 1 — Классификация тикетов

Для каждого тикета:

**Prompt:**

```
Классифицируй тикет по:

1. Тип проблемы (1-2 слова)
2. Причина (root cause)
3. Тип работ:
   - инцидент
   - запрос
   - ошибка процесса
   - ошибка системы
4. Есть ли вина команды (да/нет)
5. Если да — чья (инициатор / исполнитель / система)
```

---

## Задача 2 — Нормализация

LLM может:

* объединить похожие темы
* нормализовать хаос тегов

---

## Задача 3 — Выделение root cause

Самое ценное:

LLM может понять:

* “это не баг, это кривой процесс”
* “это ручная ошибка”
* “это системная проблема”

---

# 📊 5. Как выявить “кто косячит”

После классификации строишь:

---

## 🔥 Метрики

### 1. По командам:

* % просрочек
* средняя длительность
* возвраты
* доля “ошибок”

---

### 2. По типу проблем:

* сколько инцидентов vs запросов
* где больше всего косяков

---

### 3. Root cause breakdown:

| Причина             | %   |
| ------------------- | --- |
| ошибки пользователя | 30% |
| баг системы         | 25% |
| процесс             | 45% |

---

### 4. Heatmap:

Команда × Тип проблемы
