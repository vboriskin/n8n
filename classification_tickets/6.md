Я дам вариант, который **реально можно собрать**, а не “архитектурную фантазию”.

# Общая схема

Собираем **3 основных workflow**, не 1:

### WF-A. `tickets_ingest_and_triage`

Загрузка xlsx, очистка, признаки, rule engine, раскладка по очередям.

### WF-B. `tickets_llm_classification`

Берёт только сложные тикеты, делает LLM-классификацию, confidence gating, retry/review.

### WF-C. `tickets_canonicalize_and_aggregate`

Нормализация причин/тем и сводная аналитика.

Так устойчивее, чем один длинный workflow на 39k строк.

## 2. `Postgres — select pending llm tickets`

### SQL-логика

Выбрать, например, 200–500 pending тикетов за проход.

Лучше сразу выбирать только необработанные:

* high priority first
* high rework first
* overdue first

### Пример идеи SQL

```sql
SELECT *
FROM tickets_llm_queue
WHERE queue_status = 'pending'
ORDER BY
  CASE WHEN llm_priority = 'high' THEN 0 ELSE 1 END,
  rework_count DESC,
  overdue_flag DESC
LIMIT 300;
```

## 6. `Code — dedup representatives`

### Тип

`Code`

### Что делает

Группирует по `dedup_fingerprint`, выбирает representative.

### Выход

* 1 item на fingerprint
* плюс список duplicate IDs

### Код

```javascript
const groups = new Map();

for (const item of items) {
  const fp = item.json.dedup_fingerprint || item.json.id;
  if (!groups.has(fp)) {
    groups.set(fp, {
      representative: item.json,
      duplicate_ids: [item.json.id],
      dedup_group_size: 1
    });
  } else {
    const g = groups.get(fp);
    g.duplicate_ids.push(item.json.id);
    g.dedup_group_size += 1;
  }
}

return Array.from(groups.values()).map(g => ({
  json: {
    ...g.representative,
    duplicate_ids: g.duplicate_ids,
    dedup_group_size: g.dedup_group_size,
    representative_ticket_id: g.representative.id
  }
}));
```

```javascript
function safeJsonParse(text) {
  try {
    return JSON.parse(text);
  } catch (e) {
    const match = text.match(/\[\s*{[\s\S]*}\s*\]/);
    if (match) return JSON.parse(match[0]);
    throw e;
  }
}

function enumOr(v, allowed, fallback = 'unknown') {
  const s = String(v || '').trim();
  return allowed.includes(s) ? s : fallback;
}

const allowedRequestType = ['incident', 'service_request', 'consultation', 'data_fix', 'access', 'process_issue', 'system_defect', 'other'];
const allowedRootCauseGroup = ['system', 'process', 'human_error', 'access', 'integration', 'data_quality', 'knowledge_gap', 'unknown'];
const allowedResolutionType = ['консультация', 'исправление данных', 'ручная обработка', 'настройка', 'эскалация', 'исправление бага', 'предоставление доступа', 'без изменений', 'unknown'];
const allowedTeamFault = ['yes', 'no', 'partial', 'unknown'];
const allowedFaultStrength = ['low', 'medium', 'high', 'unknown'];
const allowedResponsible = ['initiator', 'executor_team', 'other_team', 'system', 'vendor', 'unknown'];
const allowedTri = ['yes', 'no', 'unknown'];
const allowedAutomationType = ['routing', 'data_check', 'notification', 'data_fix', 'self_service', 'decision_support', 'unknown'];
const allowedComplexity = ['low', 'medium', 'high', 'unknown'];

const raw =
  $json.message?.content ||
  $json.content ||
  $json.text ||
  $json.output_text ||
  $json.response ||
  '';

const parsed = safeJsonParse(raw);

return parsed.map(row => {
  const confidence = Number(row.confidence);
  return {
    json: {
      id: row.id || '',
      ticket_summary: row.ticket_summary || '',
      request_type: enumOr(row.request_type, allowedRequestType, 'other'),
      business_topic: row.business_topic || 'unknown',
      theme_l1: row.theme_l1 || 'unknown',
      theme_l2: row.theme_l2 || 'unknown',
      problem_theme: row.problem_theme || 'unknown',
      trigger_event: row.trigger_event || 'unknown',
      root_cause: row.root_cause || 'unknown',
      root_cause_group: enumOr(row.root_cause_group, allowedRootCauseGroup, 'unknown'),
      resolution_type: enumOr(row.resolution_type, allowedResolutionType, 'unknown'),
      team_fault_flag: enumOr(row.team_fault_flag, allowedTeamFault, 'unknown'),
      fault_signal_strength: enumOr(row.fault_signal_strength, allowedFaultStrength, 'unknown'),
      fault_evidence: row.fault_evidence || '',
      responsible_side: enumOr(row.responsible_side, allowedResponsible, 'unknown'),
      suspected_fault_team: row.suspected_fault_team || 'unknown',
      rework_flag: enumOr(row.rework_flag, allowedTri, 'unknown'),
      user_error_flag: enumOr(row.user_error_flag, allowedTri, 'unknown'),
      automation_candidate: enumOr(row.automation_candidate, allowedTri, 'unknown'),
      automation_type: enumOr(row.automation_type, allowedAutomationType, 'unknown'),
      automation_complexity: enumOr(row.automation_complexity, allowedComplexity, 'unknown'),
      automation_comment: row.automation_comment || '',
      avoidable_flag: enumOr(row.avoidable_flag, allowedTri, 'unknown'),
      failure_demand_flag: enumOr(row.failure_demand_flag, allowedTri, 'unknown'),
      decision_rationale_short: row.decision_rationale_short || '',
      confidence: Number.isFinite(confidence) ? Math.max(0, Math.min(1, confidence)) : 0,
      classification_source: 'llm_pass_1'
    }
  };
});
```

## 11. `Code — compute team_problem_score`

Если median неудобно в SQL, можно постобработкой.

### Код

Можно взять этот шаблон:

```javascript
function num(v) {
  const x = Number(v);
  return Number.isFinite(x) ? x : 0;
}

return items.map(item => {
  const j = item.json;

  const highConfFaultRate = num(j.high_conf_fault_rate);
  const reworkRate = num(j.rework_rate);
  const overdueRate = num(j.overdue_rate);
  const failureDemandRate = num(j.failure_demand_rate);
  const avoidableRate = num(j.avoidable_rate);
  const avgDuration = num(j.avg_duration_hours);
  const systemOrProcessIssueRate = num(j.system_or_process_issue_rate);

  const normalizedAvgDuration = Math.min(avgDuration / 72, 1);

  const team_problem_score =
    0.25 * highConfFaultRate +
    0.15 * reworkRate +
    0.15 * overdueRate +
    0.15 * failureDemandRate +
    0.10 * avoidableRate +
    0.10 * normalizedAvgDuration +
    0.10 * systemOrProcessIssueRate;

  return {
    json: {
      ...j,
      team_problem_score: Number(team_problem_score.toFixed(4))
    }
  };
});
```

# Что важно не делать

Вот 5 вещей, которые легко испортят результат:

### 1. Не гони все 39k сразу в LLM

Сначала triage и dedup.

### 2. Не пытайся делать “кто виноват” без confidence control

Иначе будет токсичная и недостоверная аналитика.

### 3. Не оставляй сырые root cause как есть

Иначе отчёт превратится в тысячу похожих формулировок.

### 4. Не строй аналитику только по числу тикетов

Нужны rates, а не просто counts.

### 5. Не смешивай “нормальный спрос” и “нагрузку из-за дефектов”

Поэтому `failure_demand_flag` очень важен.
