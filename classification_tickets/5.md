Погнали. Ниже — **улучшенный design n8n workflow** под твою задачу: быстро, дешевле, устойчивее и полезнее для управленческой аналитики.

---

# Цель новой версии

Не просто прогнать 39k тикетов через OpenAI, а сделать pipeline, который:

* **не тратит LLM на очевидные кейсы**
* **не гоняет дубли**
* **умеет отделять уверенные ответы от сомнительных**
* **нормализует хаос формулировок**
* **сразу готовит данные для анализа “где болит / где косячат / что автоматизировать”**

---

# Архитектура: 5 workflow

## WF1. Ingestion + preprocessing

Читает xlsx, чистит, нормализует, считает техфичи, делает дедуп-fingerprint.

## WF2. Rule-based triage

Отсекает простые кейсы без LLM и маркирует сложные.

## WF3. LLM classification

Гоняет только нужные тикеты через OpenAI, с confidence gating и retry.

## WF4. Canonicalization

Схлопывает похожие root cause / themes / team names / tags.

## WF5. Aggregation & analytics

Считает метрики команд, причин, автоматизации, повторяемости, аномалий.

---

# Общая логика прохождения тикета

```text
xlsx row
-> preprocessing
-> rule engine
   -> если simple_case = yes -> сохранить как classified_by_rule
   -> если low_info = yes -> manual_review_queue
   -> иначе -> llm_queue
-> llm pass
   -> confidence high -> accept
   -> confidence medium -> second pass
   -> confidence low -> review_queue
-> canonicalization
-> analytics
```

## Code node: derive features

Ниже скрипт, который можно ставить после твоего `normalize rows`.

```javascript
function containsAny(text, patterns) {
  if (!text) return false;
  const t = text.toLowerCase();
  return patterns.some(p => t.includes(p));
}

function bucketDuration(hours) {
  if (hours === null || hours === undefined || isNaN(hours)) return 'unknown';
  if (hours < 1) return '<1h';
  if (hours < 4) return '1-4h';
  if (hours < 24) return '4-24h';
  if (hours < 72) return '1-3d';
  return '3d+';
}

function normalizePriority(value) {
  const v = String(value || '').toLowerCase();
  if (v.includes('крит')) return 'critical';
  if (v.includes('высок')) return 'high';
  if (v.includes('сред')) return 'medium';
  if (v.includes('низ')) return 'low';
  return 'unknown';
}

const errorKeywords = [
  'ошибка', 'сбой', 'не работает', 'некорректно', 'не отображ', 'упало', 'завис',
  'не откры', 'не загру', 'не проходит', 'не удалось', 'исключение'
];

const accessKeywords = [
  'доступ', 'роль', 'права', 'разблок', 'учетн', 'авторизац', 'логин', 'пароль'
];

const dataFixKeywords = [
  'исправ', 'скоррект', 'корректиров', 'обновить данные', 'изменить данные',
  'некорректные данные', 'ошибка в данных', 'атрибут'
];

const consultKeywords = [
  'консультац', 'разъясн', 'поясн', 'уточн', 'подскаж', 'объясн'
];

const manualKeywords = [
  'вручную', 'ручн', 'оператор', 'обработан вручную', 'ручная обработка'
];

const systemKeywords = [
  'баг', 'дефект', 'ошибка системы', 'ошибка приложения', 'неработоспособность',
  'системная ошибка', 'системный сбой'
];

const integrationKeywords = [
  'интеграц', 'обмен', 'шина', 'синхрон', 'не поступил', 'не передался', 'очередь'
];

return items.map(item => {
  const j = item.json;

  const fullText = [
    j.topic, j.tag, j.description, j.resolution, j.close_code, j.service
  ].filter(Boolean).join(' ').toLowerCase();

  const descriptionLen = (j.description || '').length;
  const resolutionLen = (j.resolution || '').length;
  const textLenTotal = descriptionLen + resolutionLen;

  const has_error_keywords = containsAny(fullText, errorKeywords);
  const has_access_keywords = containsAny(fullText, accessKeywords);
  const has_data_fix_keywords = containsAny(fullText, dataFixKeywords);
  const has_consult_keywords = containsAny(fullText, consultKeywords);
  const has_manual_keywords = containsAny(fullText, manualKeywords);
  const has_system_keywords = containsAny(fullText, systemKeywords);
  const has_integration_keywords = containsAny(fullText, integrationKeywords);

  const has_rework = Number(j.rework_count || 0) > 0;
  const has_overdue = j.overdue_flag === 'yes';

  let info_quality_score = 0;
  if (j.topic) info_quality_score += 1;
  if (j.tag) info_quality_score += 1;
  if (descriptionLen > 80) info_quality_score += 2;
  else if (descriptionLen > 20) info_quality_score += 1;
  if (resolutionLen > 80) info_quality_score += 2;
  else if (resolutionLen > 20) info_quality_score += 1;
  if (j.close_code) info_quality_score += 1;

  const low_info_flag = info_quality_score < 3 ? 'yes' : 'no';

  return {
    json: {
      ...j,
      description_len: descriptionLen,
      resolution_len: resolutionLen,
      text_len_total: textLenTotal,
      has_error_keywords,
      has_access_keywords,
      has_data_fix_keywords,
      has_consult_keywords,
      has_manual_keywords,
      has_system_keywords,
      has_integration_keywords,
      has_rework,
      has_overdue,
      priority_bucket: normalizePriority(j.priority),
      duration_bucket: bucketDuration(j.total_duration_hours),
      info_quality_score,
      low_info_flag
    }
  };
});
```

## Code node: normalize org/tag fields

Это нужно, чтобы потом не было 18 вариантов одной команды.

```javascript
function norm(value) {
  return String(value || '')
    .toLowerCase()
    .replace(/ё/g, 'е')
    .replace(/[^\p{L}\p{N}\s#/_-]/gu, ' ')
    .replace(/\s+/g, ' ')
    .trim();
}

function canonicalTeam(value) {
  const v = norm(value);

  if (v.includes('hydra') || v.includes('гидра')) return 'Hydra';
  if (v.includes('поддержка hydra')) return 'Hydra';
  return value || 'unknown';
}

function canonicalExecutorGroup(value) {
  const v = norm(value);

  if (v.includes('поддержка hydra')) return 'Поддержка Hydra';
  if (v.includes('рг гидра')) return 'РГ Гидра';
  return value || 'unknown';
}

function extractTags(tagRaw) {
  const raw = String(tagRaw || '');
  const tags = raw
    .split('#')
    .map(s => norm(s))
    .filter(Boolean);

  return [...new Set(tags)];
}

function canonicalizeTag(t) {
  if (!t) return null;
  if (t.includes('it проблема') || t === 'it проблема') return 'it_problem';
  if (t.includes('перемиграц')) return 'migration_request';
  if (t.includes('доступ')) return 'access';
  return t.replace(/\s+/g, '_');
}

return items.map(item => {
  const j = item.json;

  const tags_raw = extractTags(j.tag);
  const tags_canonical = tags_raw.map(canonicalizeTag).filter(Boolean);

  return {
    json: {
      ...j,
      team_canonical: canonicalTeam(j.team),
      executor_group_canonical: canonicalExecutorGroup(j.executor_group),
      tags_raw,
      tags_canonical
    }
  };
});
```

## Code node: build fingerprint

Это база для дедупа и кластеров.

```javascript
function simplify(text) {
  return String(text || '')
    .toLowerCase()
    .replace(/ё/g, 'е')
    .replace(/\b\d{2,}\b/g, ' ') // убираем длинные числа
    .replace(/[^\p{L}\p{N}\s]/gu, ' ')
    .replace(/\s+/g, ' ')
    .trim();
}

function short(text, len) {
  return simplify(text).slice(0, len);
}

function hashCode(str) {
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    hash = ((hash << 5) - hash) + str.charCodeAt(i);
    hash |= 0;
  }
  return `fp_${Math.abs(hash)}`;
}

return items.map(item => {
  const j = item.json;

  const base = [
    short(j.topic, 120),
    short((j.tags_canonical || []).join(' '), 120),
    short(j.description, 300),
    short(j.resolution, 200),
    short(j.close_code, 80),
    short(j.executor_group_canonical, 80)
  ].join(' | ');

  const dedup_fingerprint = hashCode(base);

  return {
    json: {
      ...j,
      dedup_base: base,
      dedup_fingerprint
    }
  };
});
```

# WF2 — Rule-based triage

Тут главный буст по скорости и цене.

## Задача

Решить:

* можно ли классифицировать без LLM
* надо ли отправлять в LLM
* надо ли сразу в manual review
* какой приоритет у кейса

---

## Логика triage

### В manual review сразу:

* `low_info_flag = yes`
* совсем пустые описание/решение/теги
* аномально странные строки

### В rule-based classification:

* доступы
* чистые консультации
* очевидные data fix
* типовые ручные операции без признаков дефекта
* очевидные системные ошибки с сильными ключами

### В LLM:

* неоднозначные
* смешанные сигналы
* нужно определить ответственность
* нужен automation analysis
* сложный бизнес-контекст

---

## Code node: rule engine

```javascript
return items.map(item => {
  const j = item.json;

  let route = 'llm';
  let classified_by_rule = 'no';

  let request_type_pre = 'unknown';
  let root_cause_group_pre = 'unknown';
  let resolution_type_pre = 'unknown';
  let responsible_side_pre = 'unknown';
  let team_fault_flag_pre = 'unknown';
  let automation_candidate_pre = 'unknown';
  let triage_reason = [];

  if (j.low_info_flag === 'yes') {
    route = 'manual_review';
    triage_reason.push('low_info');
  } else {
    // access
    if (j.has_access_keywords && !j.has_error_keywords && !j.has_system_keywords) {
      route = 'rule';
      classified_by_rule = 'yes';
      request_type_pre = 'access';
      root_cause_group_pre = 'access';
      resolution_type_pre = 'предоставление доступа';
      responsible_side_pre = 'initiator';
      team_fault_flag_pre = 'no';
      automation_candidate_pre = 'yes';
      triage_reason.push('obvious_access_case');
    }

    // consultation
    else if (j.has_consult_keywords && !j.has_error_keywords && !j.has_system_keywords) {
      route = 'rule';
      classified_by_rule = 'yes';
      request_type_pre = 'consultation';
      root_cause_group_pre = 'knowledge_gap';
      resolution_type_pre = 'консультация';
      responsible_side_pre = 'initiator';
      team_fault_flag_pre = 'no';
      automation_candidate_pre = 'yes';
      triage_reason.push('obvious_consultation_case');
    }

    // obvious data fix
    else if (j.has_data_fix_keywords && !j.has_system_keywords && !j.has_error_keywords) {
      route = 'rule';
      classified_by_rule = 'yes';
      request_type_pre = 'data_fix';
      root_cause_group_pre = 'data_quality';
      resolution_type_pre = 'исправление данных';
      responsible_side_pre = 'initiator';
      team_fault_flag_pre = 'no';
      automation_candidate_pre = 'yes';
      triage_reason.push('obvious_data_fix_case');
    }

    // obvious system defect
    else if ((j.has_system_keywords || j.has_integration_keywords) && j.has_error_keywords) {
      route = 'rule';
      classified_by_rule = 'yes';
      request_type_pre = 'system_defect';
      root_cause_group_pre = j.has_integration_keywords ? 'integration' : 'system';
      resolution_type_pre = 'эскалация';
      responsible_side_pre = 'system';
      team_fault_flag_pre = 'no';
      automation_candidate_pre = 'no';
      triage_reason.push('obvious_system_case');
    }
  }

  // Повышенный приоритет на LLM
  let llm_priority = 'normal';
  if (j.has_rework || j.has_overdue) llm_priority = 'high';
  if (j.priority_bucket === 'critical' || j.priority_bucket === 'high') llm_priority = 'high';

  return {
    json: {
      ...j,
      route,
      classified_by_rule,
      request_type_pre,
      root_cause_group_pre,
      resolution_type_pre,
      responsible_side_pre,
      team_fault_flag_pre,
      automation_candidate_pre,
      triage_reason: triage_reason.join(','),
      llm_priority
    }
  };
});
```

# WF3 — LLM classification

Это уже не “всё подряд”, а только нужные тикеты.

## Поток

1. берём `route=llm`
2. дедуп по fingerprint
3. батчим representatives
4. OpenAI pass 1
5. confidence gating
6. medium confidence → second pass
7. low confidence → review
8. accepted → save
9. потом размножаем классификацию на exact duplicates

## Улучшенный system prompt

```text
Ты аналитик сервисных тикетов Service Manager.

Нужно классифицировать каждый тикет для управленческой аналитики качества процессов, причин обращений, зон ответственности, повторяемости ручных операций и кандидатов на автоматизацию.

Строгие правила:
1. Используй только входные данные.
2. Ничего не выдумывай.
3. Если данных недостаточно, ставь unknown.
4. Ответ возвращай только как JSON-массив без markdown и пояснений.
5. Для каждого тикета обязательно сохрани id.
6. ticket_summary — одно короткое предложение.
7. theme_l1 — крупная тема, theme_l2 — уточнение.
8. trigger_event — непосредственный повод обращения.
9. root_cause — глубинная причина.
10. Не ставь team_fault_flag=yes без явных или сильных косвенных признаков ошибки исполняющей стороны.
11. В спорных случаях выбирай unknown или partial.
12. suspected_fault_team указывай только если это реально следует из данных.
13. automation_candidate=yes только если кейс похож на повторяемый шаблонный ручной процесс или простой запрос, который можно частично/полностью автоматизировать.
14. confidence — число от 0 до 1.
15. decision_rationale_short — 1 короткая фраза, на чём основана классификация.
16. fault_evidence — коротко укажи сигнал, по которому сделан вывод о вине/невиновности команды.
17. failure_demand_flag=yes, если обращение возникло из-за сбоя, дефекта, ошибок процесса или ошибок данных, а не как нормальный сервисный запрос.
18. avoidable_flag=yes, если обращение потенциально можно было предотвратить улучшением процесса, качества данных, знаний или автоматизации.

Допустимые значения:
request_type = incident | service_request | consultation | data_fix | access | process_issue | system_defect | other
root_cause_group = system | process | human_error | access | integration | data_quality | knowledge_gap | unknown
resolution_type = консультация | исправление данных | ручная обработка | настройка | эскалация | исправление бага | предоставление доступа | без изменений | unknown
team_fault_flag = yes | no | partial | unknown
fault_signal_strength = low | medium | high | unknown
responsible_side = initiator | executor_team | other_team | system | vendor | unknown
rework_flag = yes | no | unknown
user_error_flag = yes | no | unknown
automation_candidate = yes | no | unknown
automation_type = routing | data_check | notification | data_fix | self_service | decision_support | unknown
automation_complexity = low | medium | high | unknown
avoidable_flag = yes | no | unknown
failure_demand_flag = yes | no | unknown

Подсказки:
- service_request: обычный запрос на обслуживание
- incident: нарушение работы сервиса
- system_defect: баг, сбой системы, дефект ПО, логики или интеграции
- data_fix: исправление данных
- access: права, роли, доступы, учетные записи
- consultation: консультация и разъяснение
- process_issue: проблема в регламенте, маршрутизации, ручном шаге, процессе
- knowledge_gap: нехватка знаний, непонимание процесса
- data_quality: неверные, неполные, испорченные данные
```

# Confidence gating

После `parse OpenAI response` делаем развилку.

## Правила

* `confidence >= 0.80` → accepted
* `0.60 <= confidence < 0.80` → second_pass
* `< 0.60` → review_queue

Отдельно:

* любой `team_fault_flag = yes` при `confidence < 0.85` → review_queue
* любой `responsible_side = executor_team` при `fault_signal_strength = low` → review_queue

---

## Code node: confidence router

```javascript
return items.map(item => {
  const j = item.json;

  let route_after_llm = 'accepted';

  if (j.confidence < 0.60) {
    route_after_llm = 'review';
  } else if (j.confidence < 0.80) {
    route_after_llm = 'second_pass';
  }

  if (j.team_fault_flag === 'yes' && j.confidence < 0.85) {
    route_after_llm = 'review';
  }

  if (j.responsible_side === 'executor_team' && j.fault_signal_strength === 'low') {
    route_after_llm = 'review';
  }

  return {
    json: {
      ...j,
      route_after_llm
    }
  };
});
```

# Second pass

Для `second_pass` не надо заново делать всё тем же prompt. Лучше узкий “refinement prompt”.

## Что уточняем

* только ambiguous fields:

  * `root_cause`
  * `responsible_side`
  * `team_fault_flag`
  * `fault_signal_strength`
  * `automation_candidate`

## System prompt second pass

```text
Ты делаешь повторную валидацию спорной классификации тикета.

Нужно уточнить только спорные поля:
- root_cause
- root_cause_group
- team_fault_flag
- fault_signal_strength
- fault_evidence
- responsible_side
- suspected_fault_team
- automation_candidate
- automation_type
- automation_complexity
- avoidable_flag
- failure_demand_flag
- confidence

Правила:
1. Используй только данные тикета и первичную классификацию.
2. Если нет уверенности, не обвиняй исполняющую команду.
3. Верни только один JSON-объект.
```

