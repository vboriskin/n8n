Да. Ниже дам вариант именно **под n8n**, с упором на:

* предфильтрацию без эмбеддингов
* few-shot
* более стабильный JSON
* снижение ложных совпадений
* удобную вставку в node

Лучше всего тут работает **2-этапная схема**:

1. **на стороне n8n** слегка нормализуем текст и, по возможности, сужаем список компетенций
2. **в LLM** даём строгий prompt с примерами и финальным JSON

Это обычно лучше, чем пытаться одной генерацией решить всё сразу.

# Вариант архитектуры в n8n

## Рекомендуемый пайплайн

### Шаг 1. Function / Code node

Подготовить:

* `normalized_name`
* `normalized_description`
* `competencies_shortlist` — если можешь сделать простой keyword-фильтр
* `competencies_full` — на всякий случай тоже оставить

### Шаг 2. LLM node

Передавать в prompt:

* исходные поля курса
* нормализованные поля
* shortlist компетенций
* 2–4 few-shot примера
* строгий JSON schema-like output

### Шаг 3. Code node

Провалидировать ответ:

* JSON parse
* все `name` входят в исходный список
* `score >= 0.7`
* нет дублей
* максимум 5 элементов

Если ответ кривой — либо retry, либо очищать программно.

---

# Почему в n8n лучше так

Потому что без эмбеддингов основная проблема — не “модель не понимает”, а:

* слишком длинный список компетенций
* шумный description
* модель начинает матчить смежные вещи

Значит, нужно:

* уменьшить пространство выбора
* дать чёткие примеры
* добавить поствалидацию

# 1. Предобработка в n8n

## Вариант Code node

Вот простой JS для n8n, который:

* нормализует текст
* собирает keywords курса
* делает грубый shortlist компетенций по пересечению слов

```javascript
const item = $input.first().json;

function normalize(text) {
  return String(text || "")
    .toLowerCase()
    .replace(/ё/g, "е")
    .replace(/[^\p{L}\p{N}\s]/gu, " ")
    .replace(/\s+/g, " ")
    .trim();
}

function tokenize(text) {
  const stopWords = new Set([
    "и","в","во","на","с","со","по","для","из","к","ко","о","об","от","до",
    "за","под","над","при","не","что","как","это","а","но","или","либо",
    "курс","обучение","программа","основы","введение","базовый","базовая",
    "базовые","нет","описания","не","указан","указана"
  ]);

  return [...new Set(
    normalize(text)
      .split(" ")
      .filter(w => w.length >= 3 && !stopWords.has(w))
  )];
}

const name = item.name || "";
const description = item.description || "";
const type = item.type || "";
const duration = item.duration || "";
const competencies = Array.isArray(item.competencies) ? item.competencies : [];

const normalized_name = normalize(name);
const normalized_description = normalize(description);
const normalized_type = normalize(type);

const courseTokens = tokenize([name, description, type].join(" "));

const scoredCompetencies = competencies.map((comp) => {
  const compText = typeof comp === "string" ? comp : (comp.name || "");
  const compNorm = normalize(compText);
  const compTokens = tokenize(compText);

  let score = 0;

  for (const token of compTokens) {
    if (courseTokens.includes(token)) score += 2;
  }

  if (normalized_name.includes(compNorm) && compNorm.length > 2) {
    score += 10;
  }

  if (normalized_description.includes(compNorm) && compNorm.length > 2) {
    score += 6;
  }

  return {
    name: compText,
    pre_score: score
  };
});

const competencies_shortlist = scoredCompetencies
  .filter(x => x.pre_score > 0)
  .sort((a, b) => b.pre_score - a.pre_score)
  .slice(0, 20)
  .map(x => x.name);

return [{
  json: {
    ...item,
    normalized_name,
    normalized_description,
    normalized_type,
    course_tokens: courseTokens,
    competencies_shortlist,
    competencies_full: competencies
  }
}];
```

---

## Что это даёт

Это не “умный” матчинг, но:

* убирает заведомо нерелевантные компетенции
* уменьшает длину prompt
* сильно снижает шанс случайных попаданий

Если shortlist пустой — в prompt можно передать первые N из полного списка или весь список.

# 2. Улучшенный prompt под n8n

Ниже версия, уже адаптированная под переменные n8n.

## Основной prompt

```text
Ты — эксперт по строгому сопоставлению учебных курсов с компетенциями.

Твоя задача — выбрать только те компетенции, которые явно и напрямую соответствуют курсу.

Используй только компетенции из переданного списка.
Запрещено:
- придумывать новые компетенции
- менять формулировки компетенций
- добавлять компетенции по косвенной или смежной связи
- возвращать что-либо кроме валидного JSON

ДОСТУПНЫЕ КОМПЕТЕНЦИИ ДЛЯ ВЫБОРА:
{{ JSON.stringify($json.competencies_shortlist?.length ? $json.competencies_shortlist : $json.competencies_full) }}

ПОЛНЫЙ СПИСОК КОМПЕТЕНЦИЙ:
{{ JSON.stringify($json.competencies_full) }}

ДАННЫЕ КУРСА:
- ID: {{ $json.id }}
- Название: {{ $json.name }}
- Описание: {{ $json.description || "Нет описания" }}
- Тип: {{ $json.type || "Не указан" }}
- Длительность: {{ $json.duration || "Не указана" }}

НОРМАЛИЗОВАННЫЕ ДАННЫЕ:
- normalized_name: {{ $json.normalized_name || "" }}
- normalized_description: {{ $json.normalized_description || "" }}
- keywords: {{ JSON.stringify($json.course_tokens || []) }}

ПРАВИЛА ОТБОРА:

1. Выбирай компетенцию только если она:
   - прямо указана в названии или описании курса
   - или является одной из основных тем курса
   - или полностью покрывает содержание курса

2. НЕ выбирай компетенцию если:
   - связь косвенная
   - это смежная область
   - компетенция может быть полезна, но не является темой курса
   - для связи нужно делать предположение
   - совпадение только по очень общему слову

3. Оценка score:
   - 0.9–1.0 — компетенция прямо указана в названии или описании почти дословно
   - 0.8–0.89 — курс в основном или полностью посвящён этой компетенции
   - 0.7–0.79 — компетенция является одной из ключевых тем курса
   - ниже 0.7 — не включать

4. reason:
   - максимум 5 слов
   - только краткий факт
   - без вводных фраз
   - без вероятностных формулировок

5. Ограничения:
   - максимум 5 компетенций
   - без дублей
   - name должен точно совпадать с элементом из полного списка компетенций
   - если нет явных совпадений, верни пустой массив

ПРИМЕР 1:
Вход:
Название: "Основы Python"
Описание: "Изучение синтаксиса Python, переменных, функций, циклов и работы с файлами"
Компетенции: ["Python", "Java", "Управление проектами", "Программирование"]

Ответ:
{
  "id": "example-1",
  "name": "Основы Python",
  "matched_competencies": [
    {
      "name": "Python",
      "score": 0.97,
      "reason": "есть в названии"
    },
    {
      "name": "Программирование",
      "score": 0.78,
      "reason": "ключевая тема курса"
    }
  ]
}

ПРИМЕР 2:
Вход:
Название: "Деловая переписка"
Описание: "Курс по оформлению писем, структуре сообщений и деловой коммуникации"
Компетенции: ["SQL", "Деловая коммуникация", "Финансовый анализ"]

Ответ:
{
  "id": "example-2",
  "name": "Деловая переписка",
  "matched_competencies": [
    {
      "name": "Деловая коммуникация",
      "score": 0.88,
      "reason": "основная тема курса"
    }
  ]
}

ПРИМЕР 3:
Вход:
Название: "Как работать в команде"
Описание: "Общий обзор взаимодействия в коллективе без привязки к конкретной компетенции"
Компетенции: ["Python", "Бюджетирование", "Информационная безопасность"]

  "name": "Как работать в команде",

Теперь обработай текущий курс.

  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",

    {
      "name": "точное название из полного списка",
```

# 3. Ещё более надёжный вариант: двухпроходный prompt

Если GigaChat иногда всё равно “перематчивает”, можно сделать **2 LLM-ноды**.

## Нода 1 — извлечение тем

Выход:

```json
{
  "id": "...",
  "name": "...",
  "topics": ["python", "функции", "циклы"],
  "strict_summary": "Базовый курс по Python"
}
```

### Prompt для первой ноды

```text
Ты анализируешь курс и выделяешь только его основные темы.

Верни только JSON.

Данные курса:
- ID: {{ $json.id }}
- Название: {{ $json.name }}
- Описание: {{ $json.description || "Нет описания" }}
- Тип: {{ $json.type || "Не указан" }}

Правила:
- topics: от 1 до 5 коротких тем
- включай только явные темы курса
- не добавляй смежные области
- strict_summary: 1 короткое предложение до 12 слов

Формат:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "topics": ["тема1", "тема2"],
  "strict_summary": "краткое описание курса"
}
```

## Нода 2 — матчинг по темам

Туда уже передаёшь `topics`, `strict_summary`, shortlist компетенций.

Это заметно стабилизирует результат.

# 4. Поствалидация в n8n

После LLM обязательно проверь результат кодом.

## Code node для валидации

```javascript
const item = $input.first().json;

let parsed;
try {
  parsed = typeof item.output === "string" ? JSON.parse(item.output) : item.output;
} catch (e) {
  throw new Error("LLM вернул невалидный JSON");
}

const originalCompetencies = Array.isArray(item.competencies_full)
  ? item.competencies_full.map(x => typeof x === "string" ? x : x.name)
  : [];

const allowed = new Set(originalCompetencies);

const seen = new Set();
const cleaned = Array.isArray(parsed.matched_competencies)
  ? parsed.matched_competencies
      .filter(x =>
        x &&
        typeof x.name === "string" &&
        allowed.has(x.name) &&
        typeof x.score === "number" &&
        x.score >= 0.7 &&
        !seen.has(x.name) &&
        seen.add(x.name)
      )
      .slice(0, 5)
      .map(x => ({
        name: x.name,
        score: Math.min(1, Math.max(0.7, Number(x.score.toFixed(2)))),
        reason: String(x.reason || "").trim().split(/\s+/).slice(0, 5).join(" ")
      }))
  : [];

return [{
  json: {
    id: parsed.id || item.id,
    name: parsed.name || item.name,
    matched_competencies: cleaned
  }
}];
```

# 5. Практические улучшения без эмбеддингов

## A. Использовать shortlist + full list одновременно

Это очень полезно.

Почему:

* shortlist сужает выбор
* full list остаётся “источником истины”
* можно требовать, чтобы `name` совпадал именно с full list

То есть модель смотрит в shortlist, но проверяется по full list.

---

## B. Few-shot примеры должны быть контрастными

Нужны не только “хорошие” совпадения, но и примеры, где надо вернуть пусто.

Лучше всего 4 типа:

1. точное совпадение
2. широкая компетенция как ключевая тема
3. несколько матчей
4. пустой результат

---

## C. Не показывать модели слишком много мусора

Если description очень длинный, лучше:

* обрезать до первых 1000–1500 символов
* убрать HTML
* убрать служебные блоки

В n8n это можно сделать заранее.

---

## D. Ввести чёрный список слишком общих слов

Если видишь частые ложные срабатывания по словам вроде:

* основы
* введение
* практика
* развитие
* современный
* эффективный

то исключай их на этапе shortlist.

---

## E. Retry strategy

Если JSON сломан:

* один повтор с более жёстким системным prompt
* либо просить модель только исправить JSON

Но лучше держать prompt коротким и структурированным.

# 8. Самый полезный концептуальный вывод

Без эмбеддингов качество тут больше зависит не от “силы модели”, а от трёх вещей:

* насколько узкий список кандидатов ты ей даёшь
* насколько жёстко запрещаешь косвенные связи
* насколько хорошо валидируешь ответ после генерации

То есть лучший буст обычно даёт не новый prompt сам по себе, а связка:

**shortlist → strict prompt → validation**
