Да. Для массового прогона я бы делал не просто “красивый prompt”, а **устойчивый production-prompt** с тремя свойствами:

1. минимальная вариативность ответа
2. защита от ложноположительных совпадений
3. встроенная самопроверка без раскрытия рассуждений

Ниже дам сразу **2 готовые версии**:

* версия для batch / n8n / Airflow
* версия с self-check внутри одного запроса

Потом покажу, какую из них лучше ставить в прод.

# 1. Версия для массового прогона (batch / n8n / Airflow)

Эта версия заточена под:

* тысячи записей
* стабильный JSON
* низкий процент “лишних” компетенций
* лёгкий постпроцессинг

Главная идея:
не просить модель “быть экспертом”, а заставить её работать как **строгий классификатор с режимом отказа**.

```text
Ты — строгий классификатор соответствия обучения и компетенций.

Задача: определить, какие компетенции из допустимого списка ЯВНО и ОДНОЗНАЧНО соответствуют обучению.

ВАЖНО:
- Используй только компетенции из списка ниже.
- Нельзя придумывать новые компетенции.
- Нельзя перефразировать названия компетенций.
- Если соответствие неочевидно, не добавляй компетенцию.
- Если информации недостаточно, не добавляй компетенцию.
- Предпочитай пустой результат ложноположительному совпадению.

ДОПУСТИМЫЕ КОМПЕТЕНЦИИ:
{{ JSON.stringify($json.competencies) }}

ДАННЫЕ ОБУЧЕНИЯ:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "description": {{ JSON.stringify($json.description || "Нет описания") }},
  "type": {{ JSON.stringify($json.type || "Не указан") }},
  "duration": {{ JSON.stringify($json.duration || "Не указана") }}
}

КРИТЕРИИ СОПОСТАВЛЕНИЯ:
Добавляй компетенцию только если выполняется хотя бы одно условие:
1. Компетенция прямо указана в названии обучения.
2. Компетенция прямо указана в описании обучения.
3. Основная тема обучения явно совпадает с компетенцией.
4. Обучение в значительной степени посвящено этой компетенции.

НЕ ДОБАВЛЯЙ компетенцию, если:
1. Совпадение только по одному общему слову.
2. Компетенция лишь косвенно связана с темой.
3. Компетенция может быть полезной, но не является явной темой курса.
4. Компетенция относится только к частному модулю, а не ко всему обучению.
5. Формулировка обучения слишком общая и не даёт уверенного основания.

ШКАЛА SCORE:
- 0.95–1.00 — компетенция прямо названа в названии обучения
- 0.90–0.94 — компетенция прямо названа в описании
- 0.80–0.89 — обучение в основном посвящено этой компетенции
- 0.70–0.79 — компетенция является значимой частью обучения

ОГРАНИЧЕНИЯ:
- Не используй score ниже 0.70
- Не используй score выше 0.95, если компетенция не названа прямо в названии
- Не используй score выше 0.90, если компетенция не названа прямо в тексте
- Максимум 5 компетенций
- Отсортируй matched_competencies по убыванию score

ПОЛЕ reason:
- максимум 5 слов
- только короткий фактический мотив
- без вводных слов
- без оценочных фраз

ПРИМЕРЫ ДОПУСТИМЫХ reason:
- "указана в названии"
- "указана в описании"
- "основная тема курса"
- "значимая часть программы"

ФОРМАТ ОТВЕТА:
Ответ должен быть только валидным JSON.
Без markdown.
Без комментариев.
Без пояснений.
Без служебного текста.

Верни строго объект:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "matched_competencies": [
    {
      "name": "точное название компетенции из списка",
      "score": 0.85,
      "reason": "краткая причина"
    }
  ]
}

Если подходящих компетенций нет, верни:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "matched_competencies": []
}
```

## Почему эта версия лучше для batch

Здесь важно не “умно рассуждать”, а **одинаково работать на 10, 1000 и 100000 строках**.

Что в ней полезно:

**Первое.** Есть явный режим консервативности:
`Предпочитай пустой результат ложноположительному совпадению.`

Это очень снижает шум в массовой обработке.

**Второе.** Есть жёсткие отрицательные критерии.
Обычно именно их не хватает, и модель начинает матчить “по ассоциации”.

**Третье.** Есть ограничение на score сверху.
Это уменьшает инфляцию уверенности.

**Четвёртое.** Есть ограничение на количество компетенций.
Для batch это полезно почти всегда.

# 2. Версия с self-check (двойная валидация в одном prompt)

Эта версия точнее, но чуть тяжелее и медленнее.
Её лучше использовать, если качество важнее скорости.

Идея такая:

* сначала модель делает черновой отбор
* потом сама же проверяет каждый кандидат по строгим правилам
* наружу отдаёт только финальный JSON

```text
Ты — строгий классификатор соответствия обучения и компетенций.

Задача: выбрать только те компетенции, которые напрямую и однозначно соответствуют обучению.

Используй только компетенции из этого списка:
{{ JSON.stringify($json.competencies) }}

ДАННЫЕ ОБУЧЕНИЯ:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "description": {{ JSON.stringify($json.description || "Нет описания") }},
  "type": {{ JSON.stringify($json.type || "Не указан") }},
  "duration": {{ JSON.stringify($json.duration || "Не указана") }}
}

ВНУТРЕННИЙ АЛГОРИТМ:
Шаг 1. Мысленно выдели основные темы обучения.
Шаг 2. Мысленно составь предварительный список компетенций-кандидатов только из допустимого списка.
Шаг 3. Для каждой компетенции-кандидата выполни самопроверку.
Шаг 4. Оставь только те компетенции, которые прошли самопроверку.
Шаг 5. Верни только итоговый JSON.

САМОПРОВЕРКА ДЛЯ КАЖДОЙ КОМПЕТЕНЦИИ-КАНДИДАТА:
Оставляй компетенцию только если на все вопросы ответ "да":
1. Эта компетенция действительно есть в допустимом списке?
2. Соответствие явно следует из названия или описания обучения?
3. Компетенция относится к основной теме курса, а не к косвенной или смежной теме?
4. Эту связь можно обосновать без догадок и домысливания?
5. Уверенность не ниже 0.70?

Если хотя бы на один вопрос ответ "нет" — исключи компетенцию.

ПРАВИЛА ОТБОРА:
- Не добавляй компетенцию по ассоциации.
- Не добавляй компетенцию по одному общему слову.
- Не добавляй компетенцию, если тема упомянута косвенно.
- Не добавляй компетенцию, если она может быть полезной, но не является явным содержанием курса.
- Не добавляй компетенцию, если она относится только к частному фрагменту обучения.
- При сомнении исключай.

ШКАЛА SCORE:
- 0.95–1.00 — компетенция прямо названа в названии обучения
- 0.90–0.94 — компетенция прямо названа в описании
- 0.80–0.89 — обучение в основном посвящено компетенции
- 0.70–0.79 — компетенция является значимой частью обучения

ПРАВИЛА ДЛЯ SCORE:
- Не используй score ниже 0.70
- Не используй score выше 0.95 без прямого упоминания в названии
- Не используй score выше 0.90 без прямого упоминания в описании
- Не округляй все значения одинаково, подбирай по силе соответствия

ПОЛЕ reason:
- максимум 5 слов
- кратко и предметно
- без вводных слов
- без оценок
- только причина соответствия

ОГРАНИЧЕНИЯ:
- Максимум 5 компетенций
- Отсортируй по убыванию score
- Используй точные названия компетенций из списка

ФОРМАТ ОТВЕТА:
Только валидный JSON.
Без markdown.
Без пояснений.
Не показывай внутренние шаги, кандидатов и самопроверку.

Верни строго:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "matched_competencies": [
    {
      "name": "точное название компетенции из списка",
      "score": 0.85,

}

Если совпадений нет:
```

# Какую версию ставить в прод

Мой практический совет такой:

**Для поточного массового прогона** — первая версия.
Она дешевле, быстрее и предсказуемее.

**Для финального качества или сложных кейсов** — вторая версия с self-check.
Она лучше отсекает спорные совпадения.

Лучший production-паттерн без эмбеддингов:

* сначала гоняешь весь массив через batch-версию
* потом только “подозрительные” записи повторно через self-check версию

Например, отправлять на второй проход записи, где:

* найдено слишком много компетенций
* все score близки друг к другу
* описание слишком короткое
* название общее, вроде “Введение”, “Базовый курс”, “Обзор”
* есть риск ложного матчинга по общим словам

# 3. Ещё более надёжный вариант для n8n / Airflow: двухпольный выход

Если у тебя downstream-пайплайн допускает это, очень советую добавить техполе `status`.

Это сильно помогает в эксплуатации.

Пример:

```json
{
  "id": "123",
  "name": "Курс",
  "status": "ok",
  "matched_competencies": []
}
```

Или:

* `ok` — модель уверенно отработала
* `insufficient_data` — описание слишком бедное
* `ambiguous` — есть неоднозначность

Но это имеет смысл только если ты **готов менять контракт**.
Если контракт уже зафиксирован, лучше не трогать.

# 5. Практические советы именно для n8n / Airflow

Есть несколько мелочей, которые сильно влияют на качество в массовом запуске.

### Нормализуй вход до LLM

Перед вызовом модели лучше:

* обрезать лишние пробелы
* заменять `null` на строку
* не передавать очень длинные HTML-описания как есть
* чистить переносы и мусорные спецсимволы

### Всегда передавай поля как JSON-safe

То есть лучше так:

```text
"name": {{ JSON.stringify($json.name) }}
```

а не просто:

```text
Название: {{ $json.name }}
```

Это уменьшает поломки JSON и проблемы с кавычками.

### Добавь post-validation

После ответа модели проверяй:

* валидный ли JSON
* все ли `name` действительно входят в список компетенций
* все ли `score >= 0.7`
* нет ли дубликатов
* не больше ли 5 элементов

Если не прошло — можно делать retry на self-check prompt.

### Делай temperature как можно ниже

Для такой задачи лучше почти детерминированный режим.

# 6. Что бы я убрал из исходного промпта

Из твоего исходника я бы убрал или усилил вот что:

**“Ты эксперт”** — звучит хорошо, но в batch-потоке это хуже, чем “ты строгий классификатор”.
Эксперт склонен интерпретировать, классификатор — отсеивать.

**“если связь явная и однозначная”** — нормально, но мало.
Нужно ещё явно прописывать, когда **нельзя** матчить.

**score 0.9–1.0 — прямо указана в названии или описании** — слишком широко.
Лучше разделить:

* название
* описание
* основная тема
* значимая часть

Иначе модель лепит 0.95 почти на всё прямое упоминание.

# 7. Что я бы предложил как следующую итерацию

Самая полезная следующая доработка — не эмбеддинги, а **few-shot на 3–7 примерах**:

* 2 хороших положительных
* 2 отрицательных
* 1 спорный, где правильный ответ — пустой массив

Это обычно даёт больше пользы, чем ещё 30 строк инструкций.
