Ниже — готовый комплект для production в n8n / Airflow.

Я дам 4 части:

1. `system prompt`
2. `user prompt`
3. `output parser / validator rules`
4. `retry prompt` для невалидного JSON или плохого результата

Плюс в конце дам короткую схему пайплайна.

---

## 1) System prompt

Это основной промпт роли. Его лучше держать стабильным и не смешивать с данными курса.

```text
Ты — строгий классификатор соответствия обучения и компетенций.

Твоя задача — по данным одного учебного курса вернуть только те компетенции, которые явно и однозначно соответствуют этому курсу.

Работай строго по правилам:

1. Используй только компетенции из переданного списка.
2. Нельзя придумывать новые компетенции.
3. Нельзя менять, сокращать или перефразировать названия компетенций.
4. Добавляй компетенцию только если соответствие прямо следует из названия или описания курса.
5. Компетенция должна относиться к основной теме курса, а не к косвенной, смежной или второстепенной теме.
6. Если есть сомнение, неоднозначность или недостаток данных — не добавляй компетенцию.
7. Предпочитай пустой результат ложноположительному совпадению.
8. Не делай выводы по ассоциации, общему контексту или одному общему слову.
9. Не добавляй компетенцию, если она относится только к отдельному модулю, а не ко всему курсу.
10. Не объясняй ход рассуждений и не показывай внутреннюю проверку.

Правила score:
- 0.95–1.00 — компетенция прямо указана в названии курса
- 0.90–0.94 — компетенция прямо указана в описании курса
- 0.80–0.89 — курс в основном посвящён этой компетенции
- 0.70–0.79 — компетенция значимо представлена в курсе

Ограничения для score:
- Нельзя использовать score ниже 0.70
- Нельзя использовать score выше 0.95 без прямого указания в названии
- Нельзя использовать score выше 0.90 без прямого указания в описании

Требования к reason:
- максимум 5 слов
- только краткое основание
- без вводных слов
- без оценочных фраз
- без пунктуации в конце

Примеры допустимых reason:
- указана в названии
- указана в описании
- основная тема курса
- значимая часть программы

Формат ответа:
- только валидный JSON
- без markdown
- без комментариев
- без пояснений
- без служебного текста
- строго один JSON-объект

Структура ответа:
{
  "id": "string",
  "name": "string",
  "matched_competencies": [
    {
      "name": "string",
      "score": 0.85,
      "reason": "string"
    }
  ]
}

Если подходящих компетенций нет, верни пустой массив matched_competencies.
Максимум 5 компетенций.
Отсортируй matched_competencies по убыванию score.
Не добавляй дубликаты.
```

## 2) User prompt

Это шаблон для каждого курса. Его уже можно кормить в batch.

Лучше передавать все потенциально “грязные” поля через `JSON.stringify(...)`.

```text
Определи соответствие курса компетенциям.

Допустимые компетенции:
{{ JSON.stringify($json.competencies) }}

Данные курса:
{
  "id": {{ JSON.stringify($json.id) }},
  "name": {{ JSON.stringify($json.name) }},
  "description": {{ JSON.stringify($json.description || "Нет описания") }},
  "type": {{ JSON.stringify($json.type || "Не указан") }},
  "duration": {{ JSON.stringify($json.duration || "Не указана") }}
}

Верни результат строго в формате:
{
  "id": {{ JSON.stringify($json.id) }},
  "name": {{ JSON.stringify($json.name) }},
  "matched_competencies": [
    {
      "name": "точное название компетенции из списка",
      "score": 0.85,
      "reason": "краткая причина"
    }
  ]
}

Если совпадений нет, верни:
{
  "id": {{ JSON.stringify($json.id) }},
  "name": {{ JSON.stringify($json.name) }},
  "matched_competencies": []
}
```

## 3) Output parser / validator rules

Это не для LLM, а для твоего пайплайна.
Сильно рекомендую валидировать ответ после модели.

### Минимальные проверки

Проверяй, что:

* ответ парсится как JSON
* корень — объект
* есть поля:

  * `id`
  * `name`
  * `matched_competencies`
* `matched_competencies` — массив
* каждый элемент массива содержит:

  * `name`
  * `score`
  * `reason`

### Бизнес-валидация

Проверяй, что:

* `matched_competencies.length <= 5`
* все `name` входят в исходный список компетенций
* нет дубликатов по `name`
* все `score` — числа
* все `score >= 0.7`
* все `score <= 1.0`
* массив отсортирован по убыванию `score`
* `reason` не пустой
* `reason` содержит не больше 5 слов

### Жёсткие fail-сценарии

Считай ответ невалидным, если:

* модель добавила компетенцию вне списка
* ответ не JSON
* есть markdown
* есть текст до или после JSON
* score строкой вместо числа
* дубликаты компетенций
* больше 5 компетенций

## 4) Retry prompt

Этот промпт отправляй, если:

* JSON сломан
* есть лишний текст
* есть компетенции вне списка
* не прошла бизнес-валидация

### Retry prompt — мягкий

```text
Исправь предыдущий ответ.

Требования:
- Верни только валидный JSON
- Без markdown
- Без комментариев
- Без пояснений
- Без текста до и после JSON
- Используй только компетенции из допустимого списка
- Не добавляй компетенции вне списка
- Не добавляй дубликаты
- Максимум 5 компетенций
- Все score должны быть числами от 0.70 до 1.00
- Отсортируй matched_competencies по убыванию score
- Сохрани исходные id и name
- Если корректных совпадений нет, верни пустой массив matched_competencies

Верни только исправленный JSON-объект.
```

### Retry prompt — жёсткий

Если модель всё равно шумит, используй ещё жёстче:

```text
Предыдущий ответ не прошёл валидацию.

Сейчас нужно только восстановить корректный JSON по правилам ниже.

Обязательные правила:
1. Только один JSON-объект
2. Никакого дополнительного текста
3. Только поля:
   - id
   - name
   - matched_competencies
4. В matched_competencies каждый объект может содержать только:
   - name
   - score
   - reason
5. Используй только точные названия компетенций из входного списка
6. Если не уверен — удаляй компетенцию
7. Если после исправления нет надёжных совпадений — верни пустой массив

Верни только JSON.
```

## 5) Рекомендуемый контракт ответа

Если контракт можно слегка улучшить, я бы рекомендовал вот такой строгий JSON schema-friendly вид:

```json
{
  "id": "123",
  "name": "Название курса",
  "matched_competencies": [
    {
      "name": "Компетенция 1",
      "score": 0.91,
      "reason": "указана в описании"
    }
  ]
}
```

Ничего больше не надо.

## 6) Рекомендуемая схема пайплайна в n8n / Airflow

### Вариант попроще

1. Preprocess входа
2. Вызов LLM
3. Parse JSON
4. Validate
5. Если ok — сохранить
6. Если fail — retry prompt
7. Если снова fail — отправить в manual review

### Вариант понадёжнее

1. Preprocess
2. LLM batch prompt
3. Validate
4. Если suspicious:

   * пустое описание
   * слишком общее название
   * 4–5 компетенций сразу
   * все score почти одинаковые
   * есть пограничные 0.70–0.74
5. Прогон через self-check prompt
6. Final validate
7. Save / manual review

## 7) Что считать suspicious record

Полезно для второго прохода:

* `description` очень короткое
* название вроде:

  * “Введение”
  * “Основы”
  * “Базовый курс”
  * “Обзор”
* найдено 5 компетенций
* все score в диапазоне 0.70–0.78
* reason у всех однотипный
* курс выглядит широким, а компетенции очень узкие

## 8) Нормализация входа перед LLM

Перед отправкой в модель полезно:

* удалить HTML-теги
* схлопнуть множественные пробелы
* удалить служебные символы
* обрезать слишком длинное описание
* заменить `null` на нормальные строки
* приводить строки к безопасному JSON-формату

Например:

* `description_clean`
* `name_clean`

## 10) Самый практичный совет

Для production я бы делал так:

* **основной prompt** — без self-check, для скорости
* **retry / second pass** — с self-check, только для проблемных записей

Так дешевле и обычно лучше по качеству, чем гонять весь поток через тяжёлый prompt.

## 11) Короткая версия self-check prompt для второго прохода

Её можно использовать именно как second pass:

```text
Проверь предыдущую классификацию заново и оставь только явно подтверждённые компетенции.

Правила:
- Используй только компетенции из списка
- Удаляй всё, что не подтверждается напрямую названием или описанием
- Удаляй всё, что относится не к основной теме курса
- Удаляй всё, где есть сомнение
- Если после проверки совпадений нет, верни пустой массив

Верни только валидный JSON в исходной структуре.
```
