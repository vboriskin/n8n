Хороший базовый промпт — уже достаточно строгий. Но для GigaChat (и вообще большинства LLM без эмбеддингов) можно сильно повысить точность за счёт **снижения свободы интерпретации** и **явной процедуры принятия решения**.

Вот ключевая проблема текущей версии:
модель *сама решает, что считать “явной связью”*, и это плавает.

Ниже — улучшенная версия + объяснение, почему это работает.

---

## 🔧 Улучшенный промпт (более детерминированный)

```text
Ты — эксперт по строгому сопоставлению учебных курсов с компетенциями.

Твоя задача — выбрать ТОЛЬКО те компетенции, которые явно и напрямую соответствуют обучению.

---

ДОПУСТИМЫЕ КОМПЕТЕНЦИИ (используй только их, без изменений):
{{ JSON.stringify($json.competencies) }}

---

ДАННЫЕ ОБУЧЕНИЯ:
- ID: {{ $json.id }}
- Название: {{ $json.name }}
- Описание: {{ $json.description || "Нет описания" }}
- Тип: {{ $json.type || "Не указан" }}
- Длительность: {{ $json.duration || "Не указана" }}

---

АЛГОРИТМ (обязателен к выполнению):

1. Сначала выдели ключевые темы обучения (1–5 коротких формулировок).
2. Для каждой компетенции из списка проверь:
   - есть ли ПРЯМОЕ совпадение по смыслу с темами обучения
   - совпадают ли ключевые термины или явно подразумеваемая область

3. Компетенция подходит ТОЛЬКО если:
   - она прямо отражает основную тему курса
   - или явно указана в названии/описании

4. НЕ добавляй компетенции если:
   - связь косвенная или общая
   - требуется додумывание
   - это “смежная область”

---

ОЦЕНКА УВЕРЕННОСТИ (строго по правилам):

- 0.9–1.0 → компетенция прямо указана в названии или описании (почти дословно)
- 0.8–0.89 → курс полностью посвящён этой компетенции
- 0.7–0.79 → одна из ключевых тем курса

Если < 0.7 → НЕ включать

---

ФОРМИРОВАНИЕ reason:
- максимум 5 слов
- только факт, без вводных слов
- без оценок и воды

Примеры:
✔ "есть в названии"
✔ "основная тема курса"
✔ "ключевой навык обучения"
✘ "похоже связано с темой"
✘ "вероятно используется"

---

ОГРАНИЧЕНИЯ:
- НЕ придумывай новые компетенции
- НЕ изменяй формулировки
- НЕ добавляй ничего вне списка
- НЕ возвращай дубликаты
- максимум 5 компетенций

---

ФОРМАТ ОТВЕТА (строго JSON, без markdown и текста):

Если есть совпадения:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "matched_competencies": [
    {
      "name": "точное название из списка",
      "score": 0.85,
      "reason": "краткая причина"
    }
  ]
}

Если совпадений нет:
{
  "id": "{{ $json.id }}",
  "name": "{{ $json.name }}",
  "matched_competencies": []
}
```

## 💡 Почему это работает лучше

### 1. Добавлен **алгоритм**

GigaChat сильно выигрывает, когда:

* есть шаги
* есть проверка условий

→ меньше “интуитивных” матчей

---

### 2. Убрана размытость “явной связи”

Ты заменил:

> "если связь явная"

на:

> конкретные критерии (ключевые темы, совпадение терминов)

---

### 3. Добавлен анти-оверматчинг блок

Это критично без эмбеддингов:

```
НЕ добавляй если:
- косвенно
- смежная область
- требуется додумывание
```

→ резко снижает false positives

---

### 4. Ограничение на количество (max 5)

LLM любит “накидывать всё подходящее”
→ лимит дисциплинирует

---

### 5. Принуждение к коротким reason

→ убирает галлюцинации и “рассуждения”

---

## 🚀 Дополнительные идеи (без эмбеддингов)

Если хочешь ещё точнее:

### ✅ 1. Добавь нормализацию текста

Перед подачей в модель:

* lower case
* убрать стоп-слова
* можно даже укоротить description

---

### ✅ 2. Дай примеры (few-shot)

GigaChat сильно улучшается от 2–3 примеров:

```
ПРИМЕР:

Курс: "Основы Python"
→ Python разработка (0.95)
→ Программирование (0.8)
```

---

### ✅ 3. Раздели на 2 шага (если можно)

1. извлечение тем
2. матчинг

Даже в одном промпте это уже частично сделано.

---

### ✅ 4. Отсортируй компетенции заранее

Если список большой:

* сначала фильтр по ключевым словам (на стороне кода)
* потом LLM

---

Если хочешь, могу:

* сделать версию **под n8n / Make / API**
* или ужать токены (оптимизация под прод)
* или адаптировать под конкретный список компетенций (это даёт +20–30% качества)
