## Шаг 2. Добавить новую ноду `Build Competency Items`

### Создать новую Code node

Название:
`Build Competency Items`

### Соединение

`Build Employee Context1`
→ `Build Competency Items`
→ `Score Competencies`

### Почему это обязательно

Потому что `Score Competencies` работает с `competency_items`, а выше они сейчас не создаются. Иначе scoring идёт по пустому массиву.

### Что должно быть на выходе

```json
{
  ...,
  "competency_items": [
    {
      "competency": {...},
      "competency_id": "...",
      "competency_name": "...",
      "target_level": 3,
      "self_assessment_item": {...},
      "jira_evidence": []
    }
  ]
}
```

# 4. Как правильно подключить LLM-анализ компетенций

Сейчас у вас одна из слабых точек — ветка после `Code in JavaScript3`.

## Правильная схема

`Code in JavaScript3`
→ `Loop Over Items` или batch-обработка
→ `Basic LLM Chain1`
→ `Parse/Validate Competency Result`

После этого нужно развести на valid / invalid.

---

## Вариант подключения без IF-ноды, через fallback merge

### Добавить новую логику

После `Parse/Validate Competency Result`:

* valid branch → в `Merge2`
* invalid branch → в `Fallback result` → тоже в `Merge2`

Если у вас нет IF-ноды, добавьте её.

---

## Рекомендуемая схема

`Parse/Validate Competency Result`
→ `IF Competency Result Valid?`

Условие:

```js
{{$json.valid}} == true
```

### Ветка true

`IF Competency Result Valid? (true)`
→ `Merge2`

### Ветка false

`IF Competency Result Valid? (false)`
→ `Fallback result`
→ `Merge2`

---

## Почему это важно

Сейчас fallback написан, но не встроен в execution path. То есть при плохом JSON от LLM вы теряете competency result, хотя fallback уже есть.

# 6. Employee summary блок

### Соединение

`Collect Results`
→ `Build Employee Summary Payload`
→ `LLM Employee Summary`
→ `Parse / Validate Employee Summary`
→ `Apply Business Rules`

### Важная правка

`Apply Business Rules` сейчас возвращает только:

```js
{
  competency_analysis: {...}
}
```

и затирает весь остальной контекст 

Значит, пока вы не вставите версию с `...$json`, planning-блок ниже будет получать обрезанный item.

### После исправления

на выходе `Apply Business Rules` должно быть:

```json
{
  ...,
  "employee": {...},
  "role_matrix": {...},
  "learning_assets_pool": [...],
  "config": {...},
  "competency_analysis": {...}
}
```

# 9. Как собрать всё обратно перед финальным отчётом

Вот здесь важный момент: `Build Final Report JSON` должен получить в одном item:

* employee context
* competency_analysis
* development_plan
* manager_plan
* one2one_guide
* jira_summary
* employee_data_quality_issues
* config

Если сейчас эти ветки идут параллельно и не merge-ятся, нужно поставить merge-ноды.

---

## Рекомендуемый способ сборки

### Основа

От `Repair Plan Constraints` идёт основной item с:

* `employee`
* `role_matrix`
* `competency_analysis`
* `development_plan`
* `config`

### Потом merge manager plan

`Repair Plan Constraints`

* `Parse / Validate Manager Pla`
  → `Merge Manager Outputs`

### Потом merge one2one

`Merge Manager Outputs`

* `Parse/Validate One2One Guide`
  → `Merge Final Inputs`

### Потом

`Merge Final Inputs`
→ `Build Final Report JSON`

---

## Какой Merge использовать

Лучше:

* `Merge by Position`, если везде один item на сотрудника
* или `Merge by Fields`, если у вас несколько сотрудников и есть общий `employee.employee_id`

Если сотрудников может быть больше одного, лучше merge по `employee_id`.

# 13. Самые частые ошибки при таком переподключении

## Потеря контекста после Code node

Любая Code node, которая делает:

```js
return [{ json: { some_field: ... } }];
```

без `...$json`, может убить контекст.

## Merge не тем режимом

Если используете `Append`, а ожидали объединение полей — downstream сломается.

## LLM-ветка работает по старому payload-контракту

Старый контракт с `payload` в LLM-нодах уже ненадёжен. В одной из LLM-нод это видно прямо в prompt: `{{ JSON.stringify($json.payload) }}` 

## Несогласованная схема role matrix

Если не нормализовать `compentecy_groups` → `competency_groups` → `competencies`, planning и report блоки будут получать пустые данные.
