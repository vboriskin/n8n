Да. Ниже дам рабочий вариант для **n8n + Jira Server/Data Center 8.20.x**:

* как настроить **HTTP Request** ноду
* какой запрос дергать
* как сделать **Code** ноду, чтобы превратить задачи Jira в `jira_competency_evidence`
* как связать это с твоим WF30

Сразу важная мысль: чтобы из Jira получить evidence **по компетенциям**, нужен способ связать задачу с `competency_id`. Обычно делают одним из 4 способов:

1. через **custom field** `competency_id`
2. через **labels**
3. через **components**
4. через словарь-мэппинг в Code node по ключевым словам

Лучше всего — **custom field**.
Если его нет, тогда labels/components.

---

# Вариант архитектуры в n8n

Минимальная цепочка:

1. `Set / Input Params`
2. `HTTP Request / Jira Search`
3. `Code / Normalize Jira Issues To Evidence`
4. дальше в твой `Build Competency Payloads`

# 1. HTTP Request node для Jira Search

## Метод

`POST`

## URL

```text
https://YOUR-JIRA-HOST/rest/api/2/search
```

Для локалки может быть так:

```text
http://jira.local/rest/api/2/search
```

или

```text
https://jira.company.local/rest/api/2/search
```

---

## Аутентификация

У тебя токен, значит есть два типовых варианта.

### Вариант A. Bearer token

В `HTTP Request`:

* Authentication: `None`
* Header:

  * `Authorization: Bearer {{$json.jira_token}}`
  * `Content-Type: application/json`

### Вариант B. Basic auth c логином + токеном

Если у вас Jira настроена именно так:

* Authentication: `Basic Auth`

Но раз ты написал “токен есть”, я бы начинал с **Bearer**.

---

## Body

Включи:

* `Send Body = true`
* `Body Content Type = JSON`

Пример body:

```json
{
  "jql": "assignee = \"87654321\" AND updated >= -180d ORDER BY updated DESC",
  "startAt": 0,
  "maxResults": 100,
  "fields": [
    "summary",
    "description",
    "labels",
    "components",
    "issuetype",
    "status",
    "resolution",
    "created",
    "updated",
    "comment"
  ]
}
```

## Если нужен custom field

Если в Jira есть поле вроде `customfield_12345`, добавь его в `fields`:

```json
{
  "jql": "assignee = \"87654321\" AND updated >= -180d ORDER BY updated DESC",
  "startAt": 0,
  "maxResults": 100,
  "fields": [
    "summary",
    "description",
    "labels",
    "components",
    "issuetype",
    "status",
    "resolution",
    "created",
    "updated",
    "comment",
    "customfield_12345"
  ]
}
```

# 2. Готовый body через expression в n8n

Если у тебя employee login приходит из предыдущей ноды:

```json
{
  "employee": {
    "employee_id": "87654321",
    "full_name": "суслов_дмитрий_сергеевич"
  }
}
```

то в HTTP node можно поставить body через expression:

```javascript
={{
{
  jql: `assignee = "${$json.employee.employee_id}" AND updated >= -180d ORDER BY updated DESC`,
  startAt: 0,
  maxResults: 100,
  fields: [
    "summary",
    "description",
    "labels",
    "components",
    "issuetype",
    "status",
    "resolution",
    "created",
    "updated",
    "comment"
  ]
}
}}
```

Если assignee у вас не employee_id, а логин/email — подставляешь нужное поле.

# 4. Code node: `Normalize Jira Issues To Evidence`

Эта нода:

* берет результат из Jira Search
* достает `issues`
* превращает их в
  `evidence_input.jira_competency_evidence`

Я дам версию с 3 режимами:

* сначала смотрим `customfield_12345`
* если нет — смотрим `labels`
* если нет — словарь `keyword -> competency_id`

## Код

```javascript
// n8n Code node
// Mode: Run Once for All Items

function asArray(v) {
  return Array.isArray(v) ? v : [];
}

function text(v) {
  return String(v ?? '').trim();
}

function lower(v) {
  return text(v).toLowerCase();
}

function uniq(arr) {
  return [...new Set(arr.filter(Boolean))];
}

function cut(str, max = 500) {
  const s = text(str);
  if (s.length <= max) return s;
  return s.slice(0, max) + '...';
}

// ---- настройки мэппинга ----

// 1) если есть custom field с competency_id
const COMPETENCY_CUSTOM_FIELD = 'customfield_12345'; 
// если нет такого поля, оставь как есть - код просто fallback'нется

// 2) labels, которые напрямую равны competency_id
const DIRECT_LABEL_MODE = true;

// 3) fallback-словарь по ключевым словам
const keywordToCompetency = {
  api: 'api_testing',
  rest: 'api_testing',
  swagger: 'api_testing',
  graphql: 'api_testing',
  websocket: 'api_testing',

  sql: 'databases',
  database: 'databases',
  db: 'databases',
  postgres: 'databases',
  oracle: 'databases',

  requirement: 'work_with_requirements',
  acceptance: 'work_with_requirements',
  traceability: 'work_with_requirements',

  defect: 'defect_management',
  bug: 'defect_management',
  regression: 'defect_management',

  stakeholder: 'communication_with_stakeholders',
  communication: 'communication_with_stakeholders',
  согласование: 'communication_with_stakeholders',
  коммуникация: 'communication_with_stakeholders'
};

// Можно передать каталог компетенций из входа
const input = items[0].json;

// ожидаем, что в item уже есть Jira response
const jiraResponse = input;
const issues = asArray(jiraResponse.issues);

// если хочешь проверять только валидные competency_id из каталога
const validCompetencyIds = new Set(
  asArray(input.competency_catalog?.competencies).map(c => c.competency_id)
);

function extractCommentText(commentField) {
  const comments = asArray(commentField?.comments);
  return comments
    .slice(0, 3)
    .map(c => text(c.body))
    .filter(Boolean)
    .join('\n');
}

function detectCompetencyIds(issue) {
  const fields = issue.fields ?? {};
  const found = [];

  // 1. custom field
  const cfValue = fields[COMPETENCY_CUSTOM_FIELD];
  if (cfValue) {
    if (Array.isArray(cfValue)) {
      for (const v of cfValue) {
        if (typeof v === 'string') found.push(v);
        else if (v?.value) found.push(v.value);
        else if (v?.name) found.push(v.name);
      }
    } else if (typeof cfValue === 'string') {
      found.push(cfValue);
    } else if (cfValue?.value) {
      found.push(cfValue.value);
    } else if (cfValue?.name) {
      found.push(cfValue.name);
    }
  }

  // 2. labels as direct competency ids
  if (DIRECT_LABEL_MODE) {
    for (const label of asArray(fields.labels)) {
      found.push(label);
    }
  }

  // 3. keyword fallback
  if (found.length === 0) {
    const haystack = lower([
      fields.summary,
      fields.description,
      extractCommentText(fields.comment),
      ...asArray(fields.labels),
      ...asArray(fields.components).map(c => c?.name)
    ].join(' '));

    for (const [keyword, competencyId] of Object.entries(keywordToCompetency)) {
      if (haystack.includes(keyword)) {
        found.push(competencyId);
      }
    }
  }

  let normalized = uniq(found.map(v => text(v)));

  // если каталог передан - отфильтруем только известные id
  if (validCompetencyIds.size > 0) {
    normalized = normalized.filter(id => validCompetencyIds.has(id));
  }

  return normalized;
}

function calcEvidenceStrength(issue) {
  const fields = issue.fields ?? {};
  const status = lower(fields.status?.name);
  const summary = lower(fields.summary);
  const desc = lower(fields.description);
  const commentText = lower(extractCommentText(fields.comment));

  let score = 0;

  if (status.includes('done') || status.includes('closed') || status.includes('resolved')) {
    score += 2;
  }

  if (summary || desc) score += 1;
  if (commentText) score += 1;

  const richSignals = [
    'провер',
    'тест',
    'валидац',
    'регресс',
    'локализ',
    'acceptance',
    'api',
    'sql',
    'schema',
    'bug',
    'дефект'
  ];

  const haystack = `${summary} ${desc} ${commentText}`;
  if (richSignals.some(s => haystack.includes(s))) {
    score += 1;
  }

  if (score >= 4) return 'strong';
  if (score >= 2) return 'medium';
  return 'weak';
}

function calcRelevance(issue, competencyId) {
  const fields = issue.fields ?? {};
  const haystack = lower([
    fields.summary,
    fields.description,
    extractCommentText(fields.comment),
    ...asArray(fields.labels),
    ...asArray(fields.components).map(c => c?.name)
  ].join(' '));

  if (haystack.includes(lower(competencyId))) return 0.95;

  const mappedKeywords = Object.entries(keywordToCompetency)
    .filter(([, compId]) => compId === competencyId)
    .map(([keyword]) => keyword);

  const hits = mappedKeywords.filter(k => haystack.includes(k)).length;

  if (hits >= 2) return 0.9;
  if (hits === 1) return 0.75;
  return 0.6;
}

const jira_competency_evidence = [];

for (const issue of issues) {
  const fields = issue.fields ?? {};
  const competencyIds = detectCompetencyIds(issue);

  if (competencyIds.length === 0) continue;

  const commentText = extractCommentText(fields.comment);

  for (const competencyId of competencyIds) {
    jira_competency_evidence.push({
      competency_id: competencyId,
      evidence_id: `JIRA-${issue.key}`,
      source: 'jira',
      issue_key: issue.key,
      title: text(fields.summary) || null,
      summary: cut([
        text(fields.summary),
        text(fields.description),
        commentText
      ].filter(Boolean).join(' | '), 700),
      relevance: calcRelevance(issue, competencyId),
      evidence_strength: calcEvidenceStrength(issue),
      status: fields.status?.name ?? null,
      issue_type: fields.issuetype?.name ?? null,
      resolution: fields.resolution?.name ?? null,
      labels: asArray(fields.labels),
      components: asArray(fields.components).map(c => c?.name).filter(Boolean),
      created: fields.created ?? null,
      updated: fields.updated ?? null
    });
  }
}

return [
  {
    json: {
      evidence_input: {
        jira_competency_evidence
      }
    }
  }
];
```

# Как это подключить в workflow

## Вариант 1. Jira и каталог уже в одном item

Если до HTTP ноды у тебя уже есть:

* `employee`
* `competency_catalog`

то можно сделать так:

### Узлы

1. `Set Input`
2. `HTTP Request / Jira Search`
3. `Merge`

   * Input 1: исходный item с employee/catalog
   * Input 2: jira response
4. `Code / Normalize Jira Issues To Evidence`

Но тут важно, чтобы `Normalize Jira Issues To Evidence` видел и:

* `issues`
* `competency_catalog`

---

## Вариант 2. Проще: сначала Jira, потом merge с основным input

Это обычно удобнее.

### Поток A

1. `Set Input`
   тут:

   * employee
   * competency_catalog
   * assessment_input

### Поток B

2. `HTTP Request / Jira Search`

### Потом

3. `Merge`
4. `Code / Build Full Input`
5. `Code / Normalize Jira Issues To Evidence`
6. `Code / Build Competency Payloads`

---

# Практический способ merge

После `HTTP Request` обычно ответ — это один item с полем `issues`.

Если хочешь просто объединить с основным input, можно через `Merge` node в режиме:

* `Combine`
* `Merge By Position`

Если в обоих потоках по одному item — это самый простой путь.

# 5. Готовый узел `Build Full Input After Jira`

Если после Merge у тебя структура некрасивая, вот промежуточная Code node, которая соберет всё в один объект.

```javascript
// n8n Code node
// Mode: Run Once for All Items

const merged = items[0].json;

return [
  {
    json: {
      employee: merged.employee,
      workflow_config: merged.workflow_config,
      competency_catalog: merged.competency_catalog,
      assessment_input: merged.assessment_input,
      scoring_input: merged.scoring_input,
      issues: merged.issues || []
    }
  }
];
```

Тогда `Normalize Jira Issues To Evidence` сможет читать:

* `competency_catalog`
* `issues`

# 7. Рекомендованный JQL

Для evidence по сотруднику обычно хорошо работают такие варианты.

## Все задачи за 6 месяцев

```sql
assignee = "87654321" AND updated >= -180d ORDER BY updated DESC
```

## Только закрытые

```sql
assignee = "87654321" AND statusCategory = Done AND updated >= -180d ORDER BY updated DESC
```

## Только QA-проекты

```sql
project in (QA, TEST) AND assignee = "87654321" AND updated >= -180d ORDER BY updated DESC
```

## С комментариями / активностью

Если у вас есть workflow через comments/history, можно отдельно искать по проектам и статусам, а потом уже фильтровать в коде.

# 8. Что я бы советовал на практике

Самый надежный путь:

* завести в Jira **labels = competency_id**
  например:

  * `api_testing`
  * `databases`
  * `defect_management`

Тогда весь mapping становится почти бесплатным:

* не нужен NLP
* не нужен keyword fallback
* evidence гораздо чище

# 9. Что нужно проверить сразу

После первого теста посмотри 3 вещи:

1. как именно у вас работает токен:

   * `Bearer`
   * или `Basic`

2. есть ли доступ к:

   ```text
   /rest/api/2/search
   ```

3. в каком поле реально можно хранить `competency_id`:

   * labels
   * custom field
   * components
