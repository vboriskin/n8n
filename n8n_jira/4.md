Ниже — нормальный **input contract** для WF30 без эвристик и без “угадывания по тексту”. Он сделан под твою задачу: взять матрицу + самооценку + Jira evidence, посчитать приоритет, прогнать top компетенции через LLM и собрать итог. Структура матрицы у тебя действительно иерархическая: роль → группы → компетенции → уровни 

## Главная идея

Нужно разделить данные на 4 слоя:

1. `employee` — кто оценивается
2. `competency_catalog` — нормализованный справочник компетенций
3. `assessment_input` — самооценка сотрудника
4. `evidence_input` — факты из Jira и других источников

Тогда код в n8n становится тупым и предсказуемым:

* не ищет soft/hard по словам
* не пытается угадать `competency_id`
* не парсит хаотичные поля
* просто делает merge по `competency_id`

# Почему именно так

## 1. `competency_catalog` должен быть уже нормализован

Это самый важный кусок.

Не надо тащить в WF30 сырую матрицу в том виде, как она хранится в большом JSON. Лучше до WF30 один раз подготовить плоский каталог:

```json
{
  "competency_id": "api_testing",
  "name": "Тестирование API",
  "skill_type": "hard",
  "priority_weight": 3,
  "criticality": "high",
  "levels": [...]
}
```

То есть WF30 должен работать не с “сырым справочником”, а с уже очищенным слоем данных.

## 2. `competency_id` обязателен везде

Вообще везде:

* в каталоге
* в самооценке
* в evidence
* в LLM output

Без этого дальше начинается ад с matching по имени.

Хорошо:

```json
"competency_id": "api_testing"
```

Плохо:

```json
"competency_name": "тестирование_api"
```

Имя можно менять. `competency_id` — нет.

## 3. `skill_type` должен быть явным

Не через маркеры. Не через regex. Не через “похоже на soft”.

Просто:

```json
"skill_type": "hard"
```

или

```json
"skill_type": "soft"
```

## 4. `priority_weight` должен приезжать готовым

Если важность компетенции уже известна бизнесом, не надо ее вычислять в коде.

Например:

* архитектура = 3
* API testing = 3
* коммуникация = 2
* документация = 2
* nice-to-have = 1

# Как лучше описать target level

Есть 3 нормальных варианта.

## Вариант A. target по грейду

Самый правильный.

```json
"employee": {
  "grade_id": "senior",
  "target_grade_id": "senior_high"
}
```

И отдельно справочник ожиданий:

```json
"target_level_matrix": {
  "manual_qa_engineer": {
    "senior": {
      "api_testing": 3,
      "sql": 2,
      "communication": 2
    },
    "senior_high": {
      "api_testing": 4,
      "sql": 3,
      "communication": 3
    }
  }
}
```

Это лучший вариант, если у тебя есть карьерная модель.

## Вариант B. target в самой компетенции

Проще.

```json
{
  "competency_id": "api_testing",
  "default_target_by_grade": {
    "middle": 2,
    "senior": 3,
    "senior_high": 4
  }
}
```

## Вариант C. target уже рассчитан до WF30

Самый тупой и надежный для самого WF30.

```json
"assessment_input": {
  "self_assessments": [
    {
      "competency_id": "api_testing",
      "self_level": 2,
      "target_level": 3,
      "comment": "..."
    }
  ]
}
```

Если хочешь простой pipeline — я бы для начала взял именно C.

# Как должен выглядеть `assessment_input`

Минимум:

```json
{
  "self_assessments": [
    {
      "competency_id": "api_testing",
      "self_level": 2,
      "comment": "Уверенно работаю с REST API"
    }
  ]
}
```

Лучше добавить:

```json
{
  "self_assessments": [
    {
      "competency_id": "api_testing",
      "self_level": 2,
      "target_level": 3,
      "comment": "Уверенно работаю с REST API",
      "confidence": "medium",
      "updated_at": "2026-03-31"
    }
  ]
}
```

Тогда можно различать:

* самооценка без комментария
* самооценка с комментарием
* самооценка, в которой сам сотрудник не очень уверен

# Как должен выглядеть `evidence_input`

Вот тут часто делают кашу. Лучше сразу привести evidence к общему формату.

## Нормальный формат одного evidence

```json
{
  "competency_id": "api_testing",
  "evidence_id": "JIRA-QA-123",
  "source": "jira",
  "issue_key": "QA-123",
  "title": "Добавлены API тесты для auth flow",
  "summary": "Покрыты позитивные и негативные сценарии",
  "evidence_strength": "strong",
  "relevance": 0.9
}
```

## Почему так

Потому что LLM потом проще кормить уже очищенными кусками, а не сырыми тикетами.

Я бы держал такие поля:

* `competency_id`
* `source`
* `title`
* `summary`
* `relevance`
* `evidence_strength`

Опционально:

* `date`
* `url`
* `status`
* `reviewed_by`
* `evidence_type`

Например:

```json
{
  "competency_id": "api_testing",
  "source": "jira",
  "evidence_type": "delivery",
  "title": "Добавлены API smoke tests",
  "summary": "Сотрудник сам написал тестовые сценарии и участвовал в валидации контрактов",
  "relevance": 0.95,
  "evidence_strength": "strong",
  "date": "2026-02-15"
}
```

# Лучший shape для внутреннего item перед LLM

Вот это, по сути, главный рабочий объект:

```json
{
  "competency_id": "api_testing",
  "competency": {
    "name": "Тестирование API",
    "group_name": "Тестирование API",
    "skill_type": "hard",
    "priority_weight": 3,
    "criticality": "high",
    "levels": [
      { "level": 0, "description": "..." },
      { "level": 1, "description": "..." },
      { "level": 2, "description": "..." },
      { "level": 3, "description": "..." },
      { "level": 4, "description": "..." }
    ]
  },
  "self_assessment": {
    "self_level": 2,
    "comment": "Уверенно тестирую REST API"
  },
  "jira_evidence": [
    {
      "issue_key": "QA-123",
      "title": "Добавлены API тесты",
      "summary": "Покрыты auth и validation сценарии",
      "evidence_strength": "strong",
      "relevance": 0.9
    }
  ],
  "target_level": 3,
  "priority_score": 71,
  "priority": "high"
}
```

Если у тебя каждый item уже такой, то дальше все очень просто.

# Как считать `priority_score`, если данные уже хорошие

Тогда логика реально становится короткой:

```js
score =
  priority_weight * 10 +
  (has_self_assessment ? 20 : 0) +
  (has_comment ? 10 : 0) +
  (has_jira_evidence ? 18 : 0) +
  (has_gap ? 16 : 0) +
  (skill_type === 'hard' ? 8 : 0) -
  (skill_type === 'soft' ? 6 : 0);
```

То есть не “ищем soft skills”, а просто читаем:

```js
skill_type === 'soft'
```

# Что бы я советовал вынести до WF30

До WF30 лучше заранее сделать:

* выбор нужной роли сотрудника
* flatten матрицы
* присвоение стабильных `competency_id`
* явное заполнение `skill_type`
* явное заполнение `priority_weight`
* вычисление `target_level`

Тогда сам WF30 превращается почти в тупой orchestration:

* merge
* score
* top N
* LLM per competency
* LLM summary
* final JSON

# Самое главное правило контракта

Один и тот же `competency_id` должен без преобразований проходить через все этапы:

```text
matrix/catalog -> self assessment -> jira evidence -> scoring -> LLM -> summary -> final output
```

Если это соблюдено, весь workflow становится сильно проще и устойчивее.
