Отлично, это как раз тот уровень, где система либо становится **продуктом**, либо превращается в дорогую и медленную игрушку. Давай сделаем две вещи:

1. **Единый end-to-end JSON контракт (реально пригодный для продакшена)**
2. **Архитектура, чтобы не умереть от latency и стоимости LLM**

# 🧠 ВАЖНЫЙ ПРИНЦИП

👉 Все LLM outputs должны попадать **в конкретные поля схемы**, а не жить “сбоку”

---

# ⚡ 3. КАК НЕ УБИТЬ LATENCY И COST

Это критично. Сейчас у тебя потенциально:

* 10–20 LLM вызовов
* большие payload
* много повторных данных

Разруливаем.

---

## 🧱 3.1 Разделяем pipeline на 2 режима

### 🔹 Heavy (offline / batch)

* competency analysis
* IDP
* promotion readiness
* delta

👉 запускается:

* раз в день
* или по кнопке

---

### 🔹 Light (real-time)

* one2one
* summary
* вопросы

👉 использует **уже готовый snapshot**

---

💡 Это дает x5–x10 ускорение UX

---

## ⚡ 3.2 Убиваем лишние LLM вызовы

Сейчас у тебя:

* 1 LLM на компетенцию (x10–15)
* * summary
* * growth
* * one2one
* * etc.

👉 это дорого

---

### ✅ Решение: batching

## Вместо:

```
LLM / Analyze One Competency (×12)
```

## Делаем:

```
LLM / Analyze All Competencies (1 вызов)
```

---

### Пример:

```text
Верни JSON:

{
  "competencies": [
    { ... },
    { ... }
  ]
}
```

👉 экономия:

* latency ↓ в 5–10 раз
* cost ↓ в 5–10 раз

---

## ⚡ 3.3 Slim payload (критично)

Перед LLM:

❌ плохо:

* весь Jira issue
* весь changelog
* все comments

✅ хорошо:

* summary
* 2–3 ключевых comment snippets
* агрегированные сигналы

---

## ⚡ 3.4 Кэширование

Кэшируй:

* Jira issues (на 1–6 часов)
* normalized evidence
* competency results

👉 ключ:

```
employee_id + lookback_days + updated_max_date
```

---

## ⚡ 3.5 Детерминизм вместо LLM

Не давай LLM делать:

* coverage_map
* confidence flags
* grade gap
* signal counting

👉 это всё Code nodes

---

## ⚡ 3.6 Разделение моделей

Используй:

* 💸 дешёвую модель → для:

  * questions
  * narrative
  * summaries

* 🧠 дорогую → только для:

  * competency analysis
  * IDP

---

## ⚡ 3.7 Parallel execution в n8n

Раздели:

* competency analysis
* jira summary
* development block

👉 они могут идти параллельно

---

## ⚡ 3.8 Ограничение размера

Перед LLM добавь:

## `Code / Trim Payload`

```javascript
const MAX = 3000;

const str = JSON.stringify($json);

return [{
  json: JSON.parse(str.slice(0, MAX))
}];
```

(или умнее — обрезать массивы)

---

## ⚡ 3.9 Snapshot storage (game changer)

Сохраняй финальный JSON:

* S3 / DB / Notion / etc.

👉 тогда:

* delta почти бесплатный
* one2one без пересчета
* UI быстрый
