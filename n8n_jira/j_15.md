# 1. Финальный n8n workflow blueprint

Это не “игрушечный workflow”, а схема под прод: с батчингом, кэшем, snapshot storage и раздельными heavy/light режимами.

## Режимы

### Heavy workflow

Считает полный профиль:

* Jira ingestion
* normalization
* evidence trace
* rule scoring
* batched LLM competency analysis
* summary
* IDP
* one2one
* promotion readiness
* snapshot save

Запуск:

* по кнопке
* по cron
* по webhook

### Light workflow

Работает от готового snapshot:

* one2one refresh
* manager digest
* employee digest
* UI fetch preparation

Запуск:

* по запросу из UI
* перед 1:1
* по открытию карточки сотрудника

---

## Heavy workflow: структура нод

## A. Input + config

1. `Webhook / Start Assessment`
2. `Set / Normalize Input`
3. `Code / Resolve Feature Flags`

Вход:

```json
{
  "employee_id": "87654321",
  "force_refresh": false,
  "lookback_days": 180,
  "features": {
    "development": true,
    "one2one": true,
    "promotion": true

}
```

## B. Cache lookup

4. `Postgres / Find Fresh Snapshot`
5. `IF / Fresh Snapshot Exists`

Логика:

* если snapshot свежий и `force_refresh = false` → отдаем его
* иначе идем в heavy pipeline

Ключ кэша:

```text
employee_id + lookback_days + source_max_updated_at
```

# 2. Как именно делать batching

Вот главный принцип: LLM должен работать не по issue и не по одной компетенции, а по уже сжатым агрегатам.

## Плохо

* 1 LLM вызов на competency
* 1 LLM вызов на growth plan per competency
* 1 LLM вызов на one2one per role

## Хорошо

* 1 LLM вызов на все competencies
* 1 LLM вызов на весь IDP
* 1 LLM вызов на one2one employee
* 1 LLM вызов на one2one manager

Итого вместо 20–30 вызовов получаешь 5–7.

# 4. DB схема: Postgres и ClickHouse

Я бы делал так:

## Postgres

Основная transactional и продуктовая база:

* сотрудники
* competency catalog
* career tracks
* trainings
* snapshots
* latest views
* one2one packs

## ClickHouse

Аналитика и дешевые агрегаты:

* issue events
* worklog events
* changelog events
* historical evidence
* team-level trend analytics
* benchmark aggregates

---

## 4.1 Postgres schema

## employees

```sql
create table employees (
  employee_id text primary key,
  full_name text not null,
  email text unique not null,
  role_id text not null,
  grade_id text not null,
  manager_id text,
  team_id text,
  is_active boolean default true,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

## competency_catalogs

```sql
create table competency_catalogs (
  catalog_version text primary key,
  created_at timestamptz default now(),
  payload jsonb not null
);
```

## career_tracks

```sql
create table career_tracks (
  track_version text primary key,
  created_at timestamptz default now(),
  payload jsonb not null
);
```

## training_catalogs

```sql
create table training_catalogs (
  training_version text primary key,
  created_at timestamptz default now(),
  payload jsonb not null
);
```

## self_assessments

```sql
create table self_assessments (
  assessment_id uuid primary key,
  employee_id text not null references employees(employee_id),
  submitted_at timestamptz not null,
  source text not null default 'manual',
  payload jsonb not null
);
create index idx_self_assessments_employee_id on self_assessments(employee_id);
```

## assessment_snapshots

```sql
create table assessment_snapshots (
  snapshot_id uuid primary key,
  employee_id text not null references employees(employee_id),
  generated_at timestamptz not null,
  lookback_days int not null,
  source_max_updated_at timestamptz,
  status text not null,
  snapshot_hash text not null,
  payload jsonb not null
);
create index idx_assessment_snapshots_employee_generated on assessment_snapshots(employee_id, generated_at desc);
```

## latest_assessment_snapshots

```sql
create table latest_assessment_snapshots (
  employee_id text primary key references employees(employee_id),
  snapshot_id uuid not null references assessment_snapshots(snapshot_id),
  generated_at timestamptz not null,
  payload jsonb not null
);
```

## one2one_sessions

```sql
create table one2one_sessions (
  session_id uuid primary key,
  employee_id text not null references employees(employee_id),
  manager_id text,
  snapshot_id uuid references assessment_snapshots(snapshot_id),
  scheduled_for timestamptz,
  status text default 'draft',
  payload jsonb not null,
  created_at timestamptz default now()
);
```

## indexes по jsonb

Если UI часто лезет в readiness, summary, competencies:

```sql
create index idx_snapshots_payload_gin on assessment_snapshots using gin (payload jsonb_path_ops);
```

## 4.2 ClickHouse schema

Нужен не для “продукта на экране”, а для:

* trend analytics
* delta
* peer benchmark
* быстрых агрегатов по Jira event stream

## jira_issue_events

```sql
create table jira_issue_events
(
  employee_id String,
  issue_key String,
  issue_id String,
  event_type String,
  event_time DateTime,
  field_name String,
  old_value String,
  new_value String,
  author String,
  payload String
)
engine = MergeTree
order by (employee_id, event_time, issue_key);
```

## jira_worklog_events

```sql
create table jira_worklog_events
(
  employee_id String,
  issue_key String,
  event_time DateTime,
  author String,
  time_spent_seconds UInt32,
  comment String
)
engine = MergeTree
order by (employee_id, event_time, issue_key);
```

## evidence_facts

```sql
create table evidence_facts
(
  employee_id String,
  snapshot_id String,
  competency_id String,
  signal_type String,
  signal_weight Float32,
  issue_key String,
  event_time DateTime
)
engine = MergeTree
order by (employee_id, competency_id, event_time);
```

## assessment_metrics

```sql
create table assessment_metrics
(
  employee_id String,
  snapshot_id String,
  generated_at DateTime,
  role_id String,
  grade_id String,
  competency_id String,
  self_level Int8,
  estimated_level Int8,
  rule_based_level Int8,
  confidence String,
  coverage String,
  priority String
)
engine = MergeTree
order by (employee_id, generated_at, competency_id);
```

# 7. Как не убить себя latency и cost

Это главная часть.

## 7.1 Только 5–7 LLM вызовов на один full assessment

Идеальная цель:

* batched competencies: 1
* employee summary: 1
* jira summary: 1
* IDP: 1
* external learning: 1
* one2one employee: 1
* one2one manager: 1

Уже хорошо.

---

## 7.2 Препроцессинг только кодом

В code nodes считаем:

* signal counts
* coverage
* guardrails
* grade gaps
* priority scores
* evidence aggregation
* training matches

LLM только:

* интерпретирует
* пишет
* формулирует

---

## 7.3 Snapshot-first UX

Для UI не запускать heavy workflow каждый раз.

Правильный UX:

* пользователь открывает карточку
* API отдает latest snapshot за 50–200 мс
* если snapshot старый, рядом появляется “обновить”

---

## 7.4 Хранить compact snapshots

В Postgres храни:

* полный snapshot
* latest projection
* summary projection

Можно даже отдельную таблицу:

```sql
create table latest_assessment_views (
  employee_id text primary key,
  generated_at timestamptz not null,
  overview jsonb not null,
  competencies jsonb not null,
  idp jsonb not null,
  one2one_employee jsonb not null,
  one2one_manager jsonb not null
);
```

Это сильно ускорит API.

---

## 7.5 Сжимать вход в LLM

Перед LLM:

* max 3 snippets per competency
* max 2 strongest signals per type
* comments режем до 300–500 символов
* descriptions summarize in code first

---

## 7.6 Разделить модели по цене

Более сильная модель:

* competencies
* IDP

Более дешевая:

* summaries
* one2one
* narrative
* learning recommendations

---

## 7.7 Использовать retries только для внешних интеграций

Для LLM не надо 5 повторов.
Достаточно:

* 1 retry на parse fail
* 1 fallback with stricter schema
