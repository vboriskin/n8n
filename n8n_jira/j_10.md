# Рекомендуемый план реализации

## Этап 1. Product foundation

Сначала добавить то, что делает результат **достоверным и пригодным для использования**:

1. **Evidence Trace / Explainability**
2. **Confidence Guardrail / Auto-Red-Flag**
3. **Skill Graph / Coverage Map**

Почему сначала это:

* без explainability менеджеры не будут доверять выводам
* без confidence guardrail система будет иногда уверенно говорить ерунду на слабых данных
* coverage map нужен, чтобы различать “слабый сотрудник” и “слабые данные”

Это лучший фундамент.

---

## Этап 2. Actionability layer

Потом добавить то, что превращает анализ в **управленческий инструмент**:

4. **Growth Plan Generator**
5. **Review Questions Generator**
6. **Manager Pack / Review-Ready Output**
7. **Auto-Narrative for Self-Review**

Почему вторым слоем:

* после оценки сразу появляется “что делать дальше”
* результат можно использовать в 1:1, perf review, self-review
* у тебя появляется реальная ценность для менеджера и сотрудника

---

## Этап 3. Comparative intelligence

Потом уже добавлять более “умные” и стратегические штуки:

8. **Delta Analysis**
9. **Outlier / Hidden Talent Detector**
10. **Promotion Readiness / Grade Fit**

Почему позже:

* им нужен более стабильный базовый пайплайн
* они сильнее завязаны на качество baseline scoring
* promotion readiness особенно опасен без хороших safeguards

---

## Этап 4. Advanced org layer

И только потом:

11. **Peer Benchmark Without Ranking**

Это уже не просто workflow на одного сотрудника, а почти внутренняя платформа аналитики.

# Мой честный вердикт

## Да, реализовать все можно

Но лучше как **модульную архитектуру**:

* `foundation module`
* `actionability module`
* `comparison module`
* `org analytics module`

Тогда ты сможешь:

* включать/выключать фичи через `workflow_config`
* не ломать весь pipeline при изменении одной фичи
* тестировать результат по слоям

# Что важно архитектурно

Чтобы реально сделать все фичи и не умереть, нужны 3 правила.

## 1. Не давать LLM считать всё подряд

LLM должен:

* интерпретировать
* суммировать
* формулировать

Но не должен:

* вычислять coverage
* считать signal density
* строить confidence heuristics
* сопоставлять evidence trace

Это лучше делать кодом в `Code` нодах.

---

## 2. Нужен единый normalized contract

Надо иметь один центральный объект, например:

```json
{
  "employee": {},
  "issues_normalized": [],
  "competency_evidence": [],
  "competency_results": [],
  "coverage_map": [],
  "guardrails": {},
  "delta_analysis": {},
  "growth_plan": [],
  "manager_pack": {},
  "self_review_narrative": {}
}
```

Тогда новые фичи будут надстраиваться, а не ломать основу.

---

## 3. Нужен snapshot layer

Для `delta analysis`, `promotion readiness`, `peer benchmark` нужен слой хранения результатов:

* прошлый snapshot по сотруднику
* baseline для grade
* возможно team aggregates

То есть часть value появится только если мы начнем сохранять outputs в БД / таблицу / storage.

# Риск, о котором сразу предупрежу

Если включить всё сразу, есть риск:

* слишком длинный workflow
* много merge/join логики
* трудно понять, где именно испортился output
* большой расход LLM вызовов
* сложный JSON контракт

Поэтому я бы сделал **не один гигантский output**, а 3 слоя output:

### 1. Core assessment

* competency results
* employee summary
* jira summary

### 2. Evidence and trust

* evidence trace
* coverage map
* confidence flags
* manual review flags

### 3. Action and planning

* growth plan
* review questions
* manager pack
* self-review narrative
* delta
* promotion readiness

Так это будет намного чище.
