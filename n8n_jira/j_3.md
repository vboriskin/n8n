# 🔥 1. Отдельная ветка: Jira Summary (без компетенций)

## Архитектура

```
Jira → Normalize Evidence → Aggregate → LLM Summary (Jira-only)
```

👉 Это НЕ зависит от competency модели
👉 Это отдельный “behavior / activity layer”

---

## 📦 Что агрегировать перед LLM

Собираешь один объект:


---

## 💡 Как это интерпретировать

LLM должен понимать:

| сигнал    | значение              |
| --------- | --------------------- |
| worklog   | участие               |
| commits   | инженерная активность |
| PR        | delivery              |
| merged PR | завершение            |
| reopen    | качество              |

---

# 🧠 LLM (Jira Summary)

## System prompt

```text
Ты анализируешь активность сотрудника по данным Jira.

Правила:
- Не выдумывай данные
- Worklog = участие, но не качество
- Commits/PR = инженерный вклад
- Учитывай завершенность задач
- Если данных мало — укажи это

Верни строго JSON
```

---

## User prompt

```text
Верни JSON:

{
  "overall_summary": "string",
  "activity_level": "low|medium|high",
  "delivery_assessment": "string",
  "engineering_signal": "string",
  "risks": ["string"],
  "confidence": "low|medium|high"
}

Данные:
{{ JSON.stringify($json.jira_summary_payload) }}
```

# 🧩 Как это соединяется с основной системой

Теперь у тебя **две независимые ветки**:

## 1. Competency analysis (что человек умеет)

## 2. Jira summary (как он работает)

И ты можешь потом:

* либо объединить их в финальном отчете
* либо использовать отдельно (например, performance review)
