Да, можно. И лучше сделать не “переделку всего workflow”, а **входной adapter layer**, чтобы ваш текущий workflow умел работать и с прежним контрактом, и с этим новым JSON.

По файлу видно, что у тебя новый вход имеет такую форму:

* первый блок: `employees` + `employee_issues`
* второй блок: `matrices`
* у сотрудника есть `employee_id`, `login`, `full_name`, `position`, `team`, `role`, `grade`
* матрицы идут по `role`, внутри `compentecy_groups` → `competencies` → `levels`
* есть `skill_categories` с инструментами/скиллами 

И еще важный момент: в примере у сотрудника роль `qa_engineer`, а в матрицах есть, например, `manual_qa_engineer`, `system_analyst` и другие. То есть нужен **role mapping layer**, иначе workflow не найдет нужную матрицу 

## Что надо сделать архитектурно

Новый слой перед твоим основным workflow:

```text
Input JSON
-> Code / Detect Input Format
-> Code / Normalize Matrix Input
-> Code / Resolve Employee
-> Code / Resolve Role Matrix
-> Code / Build Unified Root Contract
-> дальше старый workflow
```

То есть старый pipeline не ломаем. Мы просто приводим новый формат к твоему рабочему контракту.

# 3. Самая важная проблема: role mapping

У тебя в `employees.role` может быть одно значение, а в `matrices.role` другое.

Например:

* employee: `qa_engineer`
* matrix: `manual_qa_engineer`

Без адаптера это сломает подбор матрицы.

### Надо ввести mapping:

```javascript
const ROLE_MAP = {
  qa_engineer: ["manual_qa_engineer", "automation_qa_engineer", "qa_engineer"],
  analyst: ["system_analyst", "business_analyst"],
  developer: ["backend_developer", "frontend_developer", "developer"]
};
```

И дальше:

* сначала ищем точное совпадение
* если нет — ищем по `ROLE_MAP`
* если нет — fallback на `employee.role`

# 4. Какие ноды добавить в n8n

## Новые ноды перед текущим workflow

### 0.1 `Code / Detect Input Format`

```javascript
const input = $json;

const isMatrixFormat =
  Array.isArray(input) &&
  input.some(x => Array.isArray(x.employees)) &&
  input.some(x => Array.isArray(x.matrices));

return [{
  json: {
    raw_input: input,
    input_format: isMatrixFormat ? "matrix_bundle_v1" : "legacy"
  }
}];
```

---

### 0.2 `Code / Normalize Matrix Bundle`

```javascript
const raw = $json.raw_input || [];

const employeesBlock = raw.find(x => Array.isArray(x.employees)) || {};
const matricesBlock = raw.find(x => Array.isArray(x.matrices)) || {};

return [{
  json: {
    input_format: $json.input_format,
    employees: employeesBlock.employees || [],
    employee_issues: employeesBlock.employee_issues || [],
    matrices: matricesBlock.matrices || []
  }
}];
```

---

### 0.3 `Code / Pick Employee`

Тут либо берешь первого сотрудника, либо фильтруешь по `employee_id`, если он приходит снаружи.

```javascript
const targetEmployeeId = $json.target_employee_id || null;
const employees = $json.employees || [];

let employee = null;

if (targetEmployeeId) {
  employee = employees.find(e => e.employee_id === targetEmployeeId);
} else {
  employee = employees[0] || null;
}

if (!employee) {
  throw new Error("Employee not found in matrix input");
}

return [{
  json: {
    ...$json,
    selected_employee: employee
  }
}];
```

---

### 0.4 `Code / Resolve Role Matrix`

```javascript
const employee = $json.selected_employee;
const matrices = $json.matrices || [];

const ROLE_MAP = {
  qa_engineer: ["qa_engineer", "manual_qa_engineer", "automation_qa_engineer"],
  system_analyst: ["system_analyst"],
  developer: ["developer", "backend_developer", "frontend_developer"]
};

const candidates = [
  employee.role,
  ...(ROLE_MAP[employee.role] || [])
];

let matrix = matrices.find(m => candidates.includes(m.role));

if (!matrix) {
  matrix = matrices.find(m => m.role === employee.role);
}

if (!matrix) {
  throw new Error(`Matrix not found for role: ${employee.role}`);
}

return [{
  json: {
    ...$json,
    selected_matrix: matrix
  }
}];
```

---

### 0.5 `Code / Build Unified Competency Catalog`

```javascript
const matrix = $json.selected_matrix;
const competencyGroups = matrix.compentecy_groups || [];
const skillCategories = matrix.skill_categories || [];

const competencies = [];

for (const group of competencyGroups) {
  for (const comp of group.competencies || []) {
    competencies.push({
      competency_id: comp.competency_name,
      competency_name: comp.competency_name,
      group_name: group.group_name,
      role_id: matrix.role,
      type: "unknown",
      levels: (comp.levels || []).map(l => ({
        level: Number(l.level),
        description: l.description || ""
      })),
      signals: {
        jira: [],
        behavioral: [],
        skill_categories: skillCategories.map(s => ({
          category: s.category,
          skills: (s.skills || []).map(x => x.skill)
        }))
      }
    });
  }
}

return [{
  json: {
    ...$json,
    unified_competency_catalog: {
      role_id: matrix.role,
      competencies,
      skill_categories: skillCategories
    }
  }
}];
```

---

### 0.6 `Code / Build Unified Root Contract`

```javascript
const e = $json.selected_employee;
const catalog = $json.unified_competency_catalog;

return [{
  json: {
    employee: {
      employee_id: e.employee_id,
      full_name: e.full_name,
      email: e.email || null,
      role_id: e.role,
      grade_id: e.grade,
      login: e.login || null,
      position: e.position || null,
      team: e.team || null
    },
    workflow_config: {
      lookback_days: 180,
      include_competencies_without_self_assessment: true,
      include_competencies_without_evidence: true,
      features: {
        development: true,
        one2one: true,
        promotion: true
      }
    },
    competency_catalog: catalog,
    assessment_input: {
      self_assessments: []
    },
    scoring_input: {
      source: "matrix_bundle_v1"
    },
    employee_issues: $json.employee_issues || []
  }
}];
```

# 5. Как встроить это в старый workflow

У тебя дальше есть две большие ветки:

* Jira-driven
* competency/development driven

Для нового формата я бы сделал так:

## если `employee_issues` пустой

а в файле сейчас он пустой: `employee_issues: []` 

то:

* пропускаем Jira ingestion
* строим workflow только на:

  * employee
  * competency_catalog
  * matrix descriptions
  * skill_categories
  * future self-assessment / trainings

То есть:

```text
Matrix Input
-> Normalize Matrix Bundle
-> Build Unified Root Contract
-> Build Synthetic Evidence / Empty Evidence
-> Build Competency Payloads
-> Analyze All Competencies
-> Build IDP
-> Build One2One
-> Build Final Snapshot
```

---

## если позже `employee_issues` начнет заполняться

тогда их тоже можно нормализовать в тот же `normalized.issues`, как и Jira issues.

# 6. Что надо поправить в логике scoring

Потому что теперь источник не только Jira.

Я бы ввел `evidence_sources`:

```json
{
  "evidence_sources": {
    "jira": true,
    "matrix": true,
    "self_assessment": false,
    "employee_issues": false
  }
}
```

И правило:

* если есть только matrix + role rubric, это не evidence of performance
* это baseline expectation model
* confidence должен быть ниже, пока нет:

  * self assessment
  * issue/activity data
  * review data

То есть matrix-only режим — это:

* хороший режим для IDP и one2one prep
* слабый режим для performance judgment

Это очень важно.

# 7. Рекомендую ввести 3 режима workflow

## Mode A — full

Есть:

* employee
* matrix
* jira
* self_assessment

Это лучший режим.

## Mode B — matrix + employee + issues

Есть:

* employee
* matrix
* employee_issues

Нормально.

## Mode C — matrix only

Есть:

* employee
* matrix
* пустые issues

Тогда output должен быть типа:

* role/grade expectation view
* learning plan
* development strategy
* one2one prep
* без сильных выводов про фактический уровень

# 10. Мой итог

Да, этот JSON **хорошо ложится** в вашу архитектуру.
Нужно всего лишь добавить **adapter слой**, который:

* выбирает сотрудника из `employees`
* подбирает матрицу по `role`
* расплющивает `compentecy_groups -> competencies -> levels`
* сохраняет `skill_categories`
* строит единый root contract под старый pipeline

И отдельно учесть, что при пустом `employee_issues` у вас должен включаться **matrix-only mode**, а не полноценная performance-оценка 

Следующим сообщением я могу сделать уже совсем прикладную вещь: **дать тебе готовый набор n8n Code nodes для этого adapter layer**, чтобы ты просто вставил их в workflow.
